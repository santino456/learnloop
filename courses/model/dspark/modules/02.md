## [m2-paradigms] 两类 draft 范式

::: concept
title: 自回归 draft：Eagle3 路线
why: 顺序生成能建模 token 间依赖，但 draft 本身也有串行成本。

Eagle3 是 2025 年代表性的自回归投机解码方法。它用一个极小的 draft 模型（通常 1 层）依次生成候选 token，并依赖目标模型的特征做条件。优点是：

- 每个 draft token 都显式依赖前面已采样的 token，序列内一致性强。
- 在代码、数学等结构化任务上接受率较高。

缺点是：

- draft 成本随候选长度线性增长。
- 当需要一次性提出长候选序列时，draft 本身会成为瓶颈。

延伸阅读：
- [EAGLE-3: Scaling up Inference Acceleration of Large Language Models via Training-Time Test](https://arxiv.org/abs/2503.01840)
:::

::: concept
title: 并行 draft：DFlash 路线
why: 一次 forward 生成整个 token 块，速度极快，但后面 token 质量会衰减。

DFlash 用块级扩散模型（block diffusion）做并行 draft，把整个候选块一次性生成。优点是：

- draft 时间与块长基本无关，常数时间产出长序列。
- 在短候选上速度优势明显。

但论文指出，并行 draft 存在一个关键问题：**suffix decay**。块内后面的 token 缺乏对前面已生成 token 的显式依赖，导致越靠后的 token 被目标模型接受的概率越低。

延伸阅读：
- [DFlash: Block Diffusion for Flash Speculative Decoding](https://arxiv.org/abs/2602.06036)
:::

## [m2-dspark-arch] DSpark 的半自回归设计

::: concept
title: 并行 backbone 负责“快”
why: 保留并行 draft 的高吞吐优势。

DSpark 首先用一个并行 backbone 在单个 forward 内生成整个候选块。这相当于继承了 DFlash 的高吞吐特性：给定前缀 token $\mathbf{x}_{<t}$，一次性产出后续 $\gamma$ 个位置的初始 logits。

论文把这一步称为 **parallel stage**。anchor token 加上若干 masked 位置进入 backbone，得到块内每个位置的 draft 分布。
:::

::: concept
title: 轻量 sequential head 负责“准”
why: 在块内引入局部顺序依赖，缓解 suffix decay。

DSpark 在并行 backbone 之上增加了一个轻量的 **sequential head**（默认是 Markov head，可选 RNN head）。它只建模块内相邻 token 之间的转移偏置：

$$
B_k(x_0, x_{<k}) \approx \log P(x_k \mid x_0, x_{<k}) - \log P(x_k \mid x_0)
$$

其中 $x_0$ 是 anchor token。这个 head 参数量很小，但给块内每个位置引入了前缀依赖，从而显著提升后面 token 的接受率。

这种“并行主干 + 局部自回归修正”的组合就是 **Semi-Autoregressive Generation**。
:::

::: evidence
claim: 在 Qwen3-4B/8B/14B 上，DSpark 的 accepted length 比 Eagle3 提升 26.7–30.9%，比 DFlash 提升 16.3–18.4%。
source: [DSpark 论文 PDF](https://github.com/deepseek-ai/DeepSpec/blob/main/DSpark_paper.pdf)
status: verified
basis: 论文 Introduction 中给出跨 Qwen3 三个尺寸的宏观平均 accepted length 对比数据。
:::

::: compare
left: Eagle3（自回归 draft）
right: DSpark（半自回归 draft）

- 依赖建模 | 全局顺序，依赖链长 | 块内局部顺序，块间并行
- Draft 速度 | 随候选长度线性增长 | 块长增加时成本增长缓慢
- Suffix decay | 较轻 | 通过 sequential head 显著缓解
- 适用长度 | 中短候选 | 中长候选
:::

## [m2-exercise] 练习：把组件和问题对应起来

::: exercise
把 DSpark 的组件与它所解决的主要问题连起来。

- 并行 backbone
- Markov sequential head
- anchor token 机制

问题：

1. draft 后面 token 质量下降
2. 每步只产 1 个 token 太慢
3. 减少 draft 输入长度同时保持预测位置数

--- answer
- 并行 backbone → 2
- Markov sequential head → 1
- anchor token 机制 → 3
---
:::

## [m2-summary] 本章小结

- 自回归 draft 准但慢，并行 draft 快但后面 token 容易变差。
- DSpark 用“并行 backbone + 轻量 sequential head”兼顾速度与质量。
- 实测在多个目标模型上 accepted length 显著优于 Eagle3 和 DFlash。
