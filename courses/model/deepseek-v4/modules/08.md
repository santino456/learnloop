## [m8-pretraining] 预训练规模

DeepSeek-V4 系列在预训练阶段使用了：

- **DeepSeek-V4-Flash**：32T token。
- **DeepSeek-V4-Pro**：33T token。

预训练完成后，两个基础模型都能原生、高效地支持 1M 长度上下文。内部评估显示，V4-Flash-Base 已经能在多数基准上超过 V3.2-Base；V4-Pro-Base 则进一步刷新了 DeepSeek 基础模型的标准。

> 来源：[DeepSeek-V4 论文](raw/DeepSeek_V4.pdf)，第 4 节与第 1 节。

## [m8-posttraining] 后训练：专家培养 + 统一蒸馏

后训练采用两阶段范式：

1. **专家培养（Specialist Training）**：在数学、代码、agent、指令遵循等领域分别训练专家模型，先用 SFT 建立基础能力，再用 GRPO 强化学习优化领域行为。
2. **统一蒸馏（On-Policy Distillation）**：把多个专家模型的能力蒸馏到一个统一模型中，学生模型通过优化反向 KL 散度向多个教师学习。

> 来源：[DeepSeek-V4 论文](raw/DeepSeek_V4.pdf)，第 5.1 节。

## [m8-results] 核心评估结果

| 能力维度 | DeepSeek-V4-Pro-Max | 关键对比 |
|----------|---------------------|----------|
| 知识 | SimpleQA、Chinese-SimpleQA 领先开源模型 | 接近 Gemini-3.1-Pro，仍有差距 |
| 推理 | 优于 GPT-5.2、Gemini-3.0-Pro | 略逊于 GPT-5.4、Gemini-3.1-Pro，约落后 3-6 个月 |
| Agent | 与 Kimi-K2.6、GLM-5.1 相当 | 略弱于顶级闭源模型 |
| 长上下文 | 1M token 场景超过 Gemini-3.1-Pro（学术基准） | 表现强劲 |

DeepSeek-V4-Flash-Max：

- 知识评估因参数规模较小而落后。
- 推理任务在给予更大思考预算时，可与 GPT-5.2 / Gemini-3.0-Pro 相当。
- Agent 任务在部分基准上与 Pro-Max 持平，但复杂高难度任务仍有差距。

> 来源：[DeepSeek-V4 论文](raw/DeepSeek_V4.pdf)，第 1 节与第 5.3 节。

## [m8-choose] 如何选择 Pro 与 Flash

::: decision
你的场景更适合 V4-Pro 还是 V4-Flash？

- A. 需要最强知识和复杂 agent 能力，预算充足 -> V4-Pro
- B. 需要高吞吐、低成本，推理任务为主 -> V4-Flash
- C. 需要 1M token 长上下文但希望最小化 KV Cache -> V4-Flash

--- perspective
依据：论文第 1 节和第 5 节的评估结果。

A 适合对能力上限敏感的场景；B 和 C 适合对效率和成本敏感、但仍需强推理或长上下文的场景。没有绝对好坏，关键是看任务对知识深度、agent 复杂度和推理成本的权衡。
---
:::

## [m8-summary] 模块小结

- V4-Pro 在知识和复杂任务上更强，V4-Flash 在效率和成本上更优。
- 后训练采用“专家培养 + 统一蒸馏”的两阶段范式。
- 长上下文是 V4 系列最突出的工程能力，推理 FLOPs 和 KV Cache 都大幅下降。
- 选型时应根据任务对能力、延迟、成本和上下文长度的需求综合判断。

::: exercise
假设你要为公司选择一个长上下文模型做“跨百份文档的知识问答 + 代码辅助”系统。团队技术负责人倾向 V4-Flash 以降低成本，产品经理倾向 V4-Pro 以获得最强能力。你会怎么建议？先写下你的主要判断依据。

--- perspective
依据：论文第 1 节、第 5.3 节和第 5.4 节的评估结果，以及本课程对 CSA/HCA 效率优势的分析。

这个系统同时涉及长上下文（跨百份文档）和代码辅助两类任务。如果文档问答以检索和总结为主，且代码辅助以常见编程任务为主，V4-Flash 在 1M token 场景下的 KV Cache 和 FLOPs 优势可以显著降低部署成本，同时其推理能力在给予足够思考预算时并不弱。如果系统需要处理高难度的知识推理或复杂 agent 流程，V4-Pro 的能力上限更值得考虑。实际决策还需要结合具体 benchmark、延迟 SLI 和成本预算做验证。
--- tradeoffs
- V4-Pro：知识、复杂 agent 和长上下文综合能力更强，但单 token 成本和 KV Cache 更高。
- V4-Flash：1M 上下文效率极高，推理任务性价比好，但高难度知识和复杂 agent 任务可能落后。
- 混合方案：对延迟敏感或简单的查询走 Flash，对高难度推理走 Pro，但会增加系统复杂度。
--- pitfalls
- 不要仅因为“参数更大”就选择 Pro，要先定义任务的能力阈值。
- 不要假设 Flash 在所有长上下文任务上都足够，高难度知识问答需要实测。
- 不要忽略推理时的思考预算：Flash 在更大思考预算下推理能力会提升，成本模型要据此调整。
:::
