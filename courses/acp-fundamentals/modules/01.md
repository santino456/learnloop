## [m1-problem] The integration problem

Imagine three editors and three coding agents. Without a shared protocol, every editor needs a custom adapter for every agent, and every agent vendor has to understand every editor's private integration surface.

That does not scale. The problem is not model intelligence. The problem is the interface between a client and an agent.

## [m1-solution] ACP's answer

ACP puts a standard contract between the client and the agent. The client can be an editor, terminal tool, or local app. The agent can be a process that receives requests, asks for tools, and returns results.

> Core idea: implement one ACP client or one ACP-compatible agent, then reuse that integration across tools that speak the same protocol.

## [m1-roles] Who benefits

- Editor and IDE developers get one client-side integration path.
- Agent developers get one server-side protocol surface.
- Users get more choice because agents and clients are less tightly coupled.

## [m1-modes] Local and remote modes

The most concrete mode is a local agent process. A client starts the agent as a subprocess and exchanges line-delimited JSON messages over standard input and output.

Remote modes use network transports. Treat details here as implementation-dependent unless verified against the current ACP documentation.

## [m1-not] What ACP is not

ACP is not a replacement for MCP. MCP is about tools and data sources available to an agent. ACP is about how a client drives an agent session.

ACP is also not automatically the best choice for every chatbot. If your app sends one HTTP request and receives one answer, a normal API call may be enough.

## [m1-summary] Module recap

- ACP standardizes client-to-agent communication.
- It reduces custom adapter work between clients and agents.
- It is strongest when the agent is local, tool-using, and session-oriented.
- It should be chosen because it simplifies integration, not because protocols are fashionable.
