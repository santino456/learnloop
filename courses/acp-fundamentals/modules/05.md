## [m5-three-lines] 三条协议线

ACP、MCP、A2A 描述的是三种不同的关系。

- **ACP**：客户端驱动 Agent。
- **MCP**：Agent 访问工具或数据源。
- **A2A**：一个 Agent 与另一个 Agent 协作。

::: checkpoint
用一句话区分 ACP、MCP、A2A 各自解决的是“谁和谁”之间的通信。

--- answer
ACP 解决客户端↔Agent；MCP 解决 Agent↔工具/数据；A2A 解决 Agent↔Agent。
---
:::

## [m5-acp] ACP：客户端到 Agent

ACP 回答的问题是：**面向用户的客户端如何与真正干活的 Agent 对话？**

它最适合会话感强、本地上下文重要、Agent 可能需要工具的场合。

## [m5-mcp] MCP：Agent 到工具

MCP 回答的问题是：**Agent 如何安全、一致地访问外部工具、文件、API 或数据？**

ACP 客户端不需要把每个 MCP 细节都暴露给用户，它通常只负责路由工具调用并返回结果。

## [m5-a2a] A2A：Agent 到 Agent

A2A 回答的问题是：**多个 Agent 如何发现彼此、委托任务、交换结果？**

只有当系统里真的有多个 Agent 角色时才需要它。不要仅仅因为一个 Agent 内部有多个步骤就引入 A2A。

::: checkpoint
如果一个编辑器通过 ACP 调用了一个“代码审查 Agent”，而这个 Agent 又通过 A2A 把任务拆给“测试 Agent”和“文档 Agent”，那么 MCP 可能出现在哪一层？

--- answer
MCP 出现在任意 Agent 需要访问外部工具或数据时，例如代码审查 Agent 读取仓库、测试 Agent 调用测试命令。
---
:::

## [m5-choose] 怎么选

问自己哪条边界最痛：

- 客户端集成很痛 → 考虑 ACP。
- 工具和数据接入很痛 → 考虑 MCP。
- 多 Agent 交接很痛 → 考虑 A2A。

## [m5-summary] 模块小结

- ACP 连接客户端和 Agent。
- MCP 连接 Agent 和工具。
- A2A 连接 Agent 和 Agent。
- 好的架构先命名边界，再选择协议。

::: exercise
你正在做一个“本地代码助手”，它需要读取当前项目文件、调用测试命令、并把结果展示在 IDE 面板里。请选择最合理的协议组合。

- A. 全部用 ACP：IDE 面板通过 ACP 直接调用文件系统和测试命令。
- B. IDE 面板通过 ACP 驱动 Agent；Agent 通过 MCP 读取文件和调用测试命令。
- C. IDE 面板直接调用 MCP；Agent 只负责显示结果。
- D. 引入 A2A，让“文件 Agent”和“测试 Agent”互相协作。

--- answer
**B** 最合理。ACP 负责 IDE 与 Agent 之间的客户端- Agent 边界；MCP 负责 Agent 与外部工具/数据之间的接入。A2A 在本场景中没有多个独立 Agent 角色，属于过度设计。
---
:::
