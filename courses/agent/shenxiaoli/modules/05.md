## [m5-communication] 为什么是 SSE

深小理的前后端通信只用了两种模式：

| 模式 | 方向 | 用途 |
|------|------|------|
| HTTP POST | 前端 → 后端 | 用户发送消息 |
| SSE (Server-Sent Events) | 后端 → 前端 | Agent 思考/回答/工具调用 流式推送 |

没有 WebSocket、没有轮询。选择 SSE 的原因：

- **单向足够**：前端只需要收 Agent 的响应，不需要持续双向通信
- **Nginx 兼容**：`proxy_buffering off` 一行配置搞定
- **协议简单**：纯 HTTP 文本流，Chrome DevTools 可以直接看
- **断线处理简单**：HTTP 连接断开 = 流结束，立即可以重试

::: checkpoint
什么场景下 SSE 不够用，必须用 WebSocket？

--- answer
当需要前端**频繁主动发送**消息且需要后端即时响应时。比如：协同编辑（每个人每次按键都推送）、实时游戏（高频双向状态同步）、语音/视频通话（需要全双工）。深小理是问答式交互，用户每次只发一条消息然后等回复，SSE 完全够用。
---
:::

## [m5-pipeline] SSE 两阶段 Pipeline

```
后端 Agent 循环 ──→ asyncio.Queue ──→ SSE Generator ──→ 前端 EventSource
                        │
                  ┌─────┴──────┐
                  │            │
              Stage 1       Stage 2
         (Agent 运行时)   (Agent 完成后)
         thinking         sentence chunks
         reasoning        (打字机效果, 16char/chunk, 25ms间隔)
         tool_calls       done (完整内容)
         tool_result
         chunk
```

**Stage 1**：实时推送。Agent 每产生一个事件（思考、推理、工具调用、文本片段），立即通过 Queue 流向 SSE。

**Stage 2**：Agent 循环结束后。最终回复被拆成句子级别的 chunk（16 字符），25ms 间隔逐块推送——制造"AI 在打字"的体验。最后发 `done` 事件带完整内容。

::: exercise
如果你想改打字机效果的参数——比如改成 8 字符 / 40ms 间隔——你会改后端的哪一行？

提示：去看 `backend/app/main.py` 中 SSE streaming 的 Stage 2 逻辑。

--- answer
需要改 `app/main.py` 中生成 sentence chunks 的参数：`chunk_size` 和 `asyncio.sleep` 的时间。具体位置在 `/chat/stream` 端点中，Agent 完成后、yield SSE events 的那段代码。
---
:::

## [m5-reducer] 前端状态管理：纯函数 Reducer

前端用了一个**纯函数状态 reducer** 来消费 SSE 事件流——和 React 的 `useReducer` 模式一致，但独立于组件生命周期：

```
SSE Event → reduceChatStreamEvent(currentState, event) → newState → React setState
```

这个 reducer 处理所有事件类型：

| 事件 | Reducer 行为 |
|------|-------------|
| `thinking` | 累加 loop counter，为新一轮 ReAct 创建 assistant placeholder |
| `reasoning` | 累加到当前 thinking 内容 |
| `tool_calls` | 在当前 assistant 后面插入 tool message（去重 by tool_call_id） |
| `tool_result` | 更新 tool 状态，确保后面有 assistant placeholder |
| `chunk` | 累加到当前 assistant 的 content |
| `ask_user` | 插入交互式 survey 组件 |
| `privacy_confirm` | 插入隐私确认对话框 |
| `done` | 标记完成，填充 message ID |
| `error` / `quota_exceeded` / `service_busy` | 终端错误状态 |

## [m5-recovery] 断流恢复：recoverSessionTurn

SSE 连接断开时（网络波动、服务器重启），前端不会丢失数据。`recoverSessionTurn()` 方法：

1. 检测到 SSE 流中断
2. 调 REST API 重新获取会话消息
3. 对比本地状态——如果这一轮已经保存了，就用服务端的
4. 无缝接上

::: checkpoint
为什么 SSE 断开后不能简单地重连 SSE 继续收？为什么必须从 REST API 重新拉？

--- answer
因为 SSE 是单向推送——断连期间的事件已经丢失了。如果只重连 SSE，Server 会从头开始推（新请求），中间丢的事件找不回来。REST API 是"拉"模式，可以获取到断连期间服务端已经写入数据库的完整消息。
---
:::

## [m5-richcontent] RichContent：从 Markdown 到 Block

`RichContent.tsx` 不是简单的 Markdown 渲染器——它是一个 **Markdown-to-Block 管线**：

```
输入: 一段混合文本 + 图片路径
  ↓
splitIntoSegments() — 按 \n\n 分段，保留代码块
  ↓
逐段解析类型
  ├── 首段有图？ → Hero block（全宽大图 + 点击放大）
  ├── > 📍 开头？ → Location card（地点卡片）
  ├── 2+ 连续链接？ → Links block（标签按钮组）
  ├── 2+ 图片？ → Gallery block（图片堆叠）
  └── 其他 → ReactMarkdown（GFM + 中文友好分行）
  ↓
mergeGalleries() — 合并相邻图库
  ↓
输出: Block 列表 → React 组件渲染
```

### 关键渲染决策

- **表格**：`overflow-x: auto` + 斑马纹 + sticky header
- **代码块**：`pre` 带深色背景 + 语言标签
- **H2 标题**：紫色左边框强调
- **图片**：`cursor: pointer` + `onClick` 在新 tab 打开原图
- **引用块**：左侧棕色边框，做地点卡片的语义标记

## [m5-oss] 图片解析：resolveImageUrl

生产环境中，图片不经过 ECS 带宽。`/kb/campus/images/...` 和 `/skills/...` 路径被重写为 OSS 直连 URL：

```
/kb/campus/images/buildings/library.jpg
  → https://shenxiaoli-assets.oss-cn-shenzhen.aliyuncs.com/campus/buildings/library.jpg

/skills/campus-info/assets/images/2026-06/map.png
  → https://shenxiaoli-assets.oss-cn-shenzhen.aliyuncs.com/skills/campus-info/assets/images/2026-06/map.png
```

开发环境中两个 base URL 为空字符串，图片从本地磁盘加载。

OSS 启用 Referer 防盗链：只允许 `*shenxiaoli.cn`，禁止空 Referer（防止直接 curl 下载）。

## [m5-ux] 交互细节：几个不在 PRD 里的设计

### Fork Turn（分叉对话）

长按用户消息 450ms 或右键，弹出"从这一轮分叉"按钮。分叉 = 复制当前会话到这条消息之前的所有历史，创建新会话。

用处：用户可以说"A 方案和 B 方案哪个好"，Agent 推荐后，用户可以叉掉，换个问法再试。

### 呼吸动画

工具组（ToolGroup）在"正在查资料"状态下有一个脉动圆点动画 (`animate-activity-breathe`)。这不是装饰——它告诉用户 Agent 正在干活，不是卡了。

### Tool Limit Warning

当 Agent 循环超过工具调用上限，一个琥珀色横幅从顶部滑入。5 秒后自动消失。对比静默失败，这个设计让用户知道发生了什么。

::: exercise
打开深小理，仔细观察这三个交互细节。然后想：如果去掉其中两个，用户体验会差在哪里？每个细节你给几分（1-10），为什么？

--- perspective
依据：基于深小理前端实际实现和用户行为观察。

这些细节的共同特点是：**花很少的代码解决了一个可能让用户困惑的问题**。Fork turn 在不破坏当前对话的前提下给用户探索空间；呼吸动画用 3 行 CSS 解决了"工具在跑还是卡了"的疑虑；tool limit warning 比静默截断体面得多。

好产品的 UX 不是功能多，是在每个用户可能困惑的节点上放一个提示。
---
:::

## [m5-summary] 模块小结

- SSE 够用——单向推送 + Nginx 兼容 + 协议简单。
- 两阶段 pipeline：Stage 1 实时推送 Agent 事件，Stage 2 打字机效果。
- 纯函数 reducer 消费 SSE 事件流，状态可追溯。
- `recoverSessionTurn` 处理断流恢复——SSE 断了用 REST API 补。
- RichContent 不是 Markdown 渲染器，是 Markdown → Block 转换管线。
- OSS 直连解放 ECS 带宽，防盗链保护资源。
- UX 细节决定体验：fork turn、呼吸动画、tool limit warning。
