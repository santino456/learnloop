## [m1-bottleneck] LLM 推理的主要瓶颈

::: concept
title: 自回归生成的串行约束
why: 每个新 token 都依赖之前所有 token，这是推理 latency 的根本来源。

大型语言模型在解码阶段采用自回归方式：生成第 $t$ 个 token 时，必须先把前 $t-1$ 个 token 的 key/value 都算好并加载。这带来两个后果：

- **计算并行度低**：虽然每次 forward 算力很大，但用户实际只拿到 1 个新 token。
- **内存墙**：随着序列变长，KV-cache 容量和带宽成为瓶颈，尤其在大 batch 高并发场景。

这也是为什么生产环境里 LLM 推理成本高、响应慢的核心原因。

延伸阅读：
- [DeepSeek-V3 Technical Report](https://arxiv.org/abs/2412.19437)（对 MoE 推理瓶颈有系统分析）
- [vLLM 论文：PagedAttention](https://arxiv.org/abs/2309.06180)（KV-cache 管理方向的经典工作）
:::

::: concept
title: 投机解码的基本思想
why: 用便宜的小模型先“猜”一串 token，再由大模型一次性验证，可以把串行变成批量。

投机解码（Speculative Decoding）的核心流程是：

1. **Draft**：用一个轻量 draft 模型一次性生成 $\gamma$ 个候选 token。
2. **Verify**：用目标大模型对这 $\gamma$ 个 token 做一次 forward，按规则接受或拒绝。
3. **Resample**：从第一个被拒绝的位置重新采样，并把已接受部分作为下一轮输入。

只要 draft 模型足够快、接受率足够高，就能把多次串行的大模型调用压缩成一次，整体吞吐显著提升。

延伸阅读：
- [Accelerating Large Language Model Decoding with Speculative Decoding](https://arxiv.org/abs/2302.01318)
- [Fast Inference from Transformers via Speculative Decoding](https://arxiv.org/abs/2211.17192)
:::

## [m1-dspark-intro] DSpark 是什么

::: evidence
claim: DSpark 是 DeepSeek 与北京大学在 2026 年 6 月底开源的投机解码框架，目标是让现有 DeepSeek-V4 推理显著提速。
source: [DeepSpec GitHub 仓库](https://github.com/deepseek-ai/DeepSpec)
status: verified
basis: 论文 PDF 与仓库 README 均标注 DeepSeek-AI、Peking University 及 MIT 协议；MarkTechPost 等报道发布时间为 2026-06-27。
:::

::: evidence
claim: 在生产环境部署中，DSpark 让 DeepSeek-V4-Flash 每用户生成速度提升 60–85%，V4-Pro 提升 57–78%。
source: [DSpark 论文 PDF](https://github.com/deepseek-ai/DeepSpec/blob/main/DSpark_paper.pdf)
status: verified
basis: 论文 Figure 7 给出在不同 SLA 下，V4-Flash 在严格延迟约束下 TPS 提升 +85%，V4-Pro 提升 +78%；宽松约束下吞吐分别提升 +51% 和 +52%。
:::

::: compare
left: 标准自回归解码
right: DSpark 投机解码

- 每步产出 | 1 个 token | 平均多个 token（接受长度决定）
- 瓶颈 | KV-cache 带宽、串行依赖 | draft 质量与验证效率
- 质量 | 与目标模型一致 | 理论上无损（保持目标分布）
- 额外成本 | 无 | 需要训练/加载 draft 模块
:::

## [m1-exercise] 练习：为什么小模型能加速大模型？

::: exercise
假设小模型生成 5 个 token 的成本只有大模型的 1/10，但大模型一次能验证 5 个 token。如果小模型的 5 个 token 中有 3 个被接受，整体能省多少大模型调用？

--- answer
原来需要 3 次大模型调用才能拿到 3 个 token；现在 1 次验证即可拿到 3 个接受 token，加上小模型成本相当于 0.1 次大模型调用。总成本约 1.1 次，近似省掉 2/3 的大模型调用。
---
:::

## [m1-summary] 本章小结

- LLM 推理慢的根源是自回归串行和 KV-cache 开销。
- 投机解码通过“草稿 + 验证”把串行变成批量。
- DSpark 不是新模型，而是给 DeepSeek-V4 加的推理加速层，已在生产 API 部署。
