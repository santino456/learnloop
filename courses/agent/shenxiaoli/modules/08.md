## [m8-why] 这个模块的目的

前面 7 个模块讲了"是什么"和"怎么做"。这个模块只问一个问题：**每个技术决策的代价是什么？如果重来一次，哪些会不一样？**

这不是复盘检讨——是判断力的训练。好工程师不只是能实现方案，而是能说清楚**为什么选 A 不选 B，以及 B 在什么场景下其实更好**。

## [m8-noodb] 为什么不用向量数据库

深小理的知识库搜索用的是自研 BM25，不是向量检索。

**选 BM25 的理由**（在当时是正确的）：
- 几百篇中文文章，词频匹配够用
- 零部署成本，不需要单独的服务进程
- 不需要 embedding API（省钱 + 省延迟）

**什么情况下这个选择会站不住脚**：
- 知识库从几百篇增长到几千篇，BM25 稀疏召回开始漏相关文档
- 用户开始用自然语言问法（"有什么好吃的" → 找食堂文章），而不是关键词匹配
- 多语言场景（中英混合查询）

::: exercise
假设深小理的知识库增长到 5000 篇，BM25 召回率下降。你有两个升级方案：
A) 把 BM25 换成向量检索（如 pgvector）
B) BM25 + 向量检索混合（先 BM25 粗排，再向量精排）

选哪个？为什么？

--- perspective
依据：目前没有实际数据支撑哪种方案更好——这是训练"做技术判断"的思路练习。

方案 A 简单但迁移成本高（需要生成所有文档的 embedding），且纯向量检索在某些精确匹配场景（如"学号前六位"）可能反而不如 BM25。

方案 B 保持 BM25 的精确匹配优势，同时用向量检索补全语义召回——但增加了系统复杂度（两套索引、两次查询）。

思考的角度应该包括：用户真实查询模式是什么样的？精确匹配和语义搜索的比例是多少？需要数据支撑才能做决定，而不是凭直觉选一个。
---
:::

## [m8-noorm] 为什么不用 ORM

**选了直接写 SQL 的理由**：
- 查询模式简单——主要是 CRUD + 少量 JOIN
- messages 存 JSONB，不需要映射到 Python 对象
- 启动更快（没有 ORM schema 反射）
- 调试更直接——能看实际 SQL

**这笔账在什么场景下会反噬**：
- 表结构变更时没有 migration 框架——现在 `CREATE TABLE IF NOT EXISTS` + 手动 ALTER
- 团队变大后，不同人对 SQL 风格理解不一致
- 如果要加软删除、审计日志、自动时间戳——需要自己写，ORMs 自带

**诚实的判断**：对于深小理当前的规模（一个人维护、10 张表），直接 SQL 是对的。如果团队变成 3 人以上且频繁改 schema，引入 Alembic（SQLAlchemy 的 migration 工具）是合理的下一步。

## [m8-lock] Session Lock 的边界

每个 session 一个 `asyncio.Lock`，同一 session 的消息串行处理。

**这个设计很巧妙**：
- 防止同一 session 的并发消息导致状态混乱
- 不同 session 之间完全并行，互不阻塞
- LRU 上限 1024 条，防止内存泄漏

**但也有代价**：
- 如果用户疯狂连续发消息，第二条要等第一条的整个 ReAct 循环跑完
- 锁的超时释放是被动的（靠 LRU 淘汰），没有主动的 TTL

::: checkpoint
什么场景下 per-session lock 会出问题？

--- answer
当一次 ReAct 循环特别慢（比如工具调用 10 次、每次 5 秒），用户在此期间发了新消息。新消息会排队等旧消息跑完才处理。虽然锁本身不会死锁（asyncio.Lock 在协程正常退出时释放），但用户体验是：发完第二条消息等了 50 秒才开始处理。

实际场景中这个概率很低——绝大多数对话 1-3 个工具调用就完成了。如果真的出现，可能需要加一个"放弃当前处理"的信号机制。
---
:::

## [m8-baremetal] 裸机部署：便利与代价的再审视

裸机部署的便利我们已经讲了（M1）。现在说说代价：

**当前可接受的代价**：
- 3-5 秒重启窗口 → 用户量小时可接受
- 手动 git pull + deploy.sh → 部署频率低（一天几次），手动可以
- 依赖 Python venv 手动管理 → 一个人维护，操作熟练

**用户量增长后的升级路径**：
1. 先在 systemd 层面做双端口 + Nginx reload（零停机重启）
2. 多 worker 的 rate limit 迁移到 Redis
3. 如果后面要拆微服务，再考虑 Docker

不急于容器化的核心判断：**Docker 解决的是环境和依赖管理问题（多服务、多环境、多人协作），而不是性能问题。** 当前问题是单机单应用，Docker 带来的额外好处不足以抵消其复杂度。

## [m8-coldhot] 冷热分层的未竟之路

冷热分层（M3）是一个很聪明的设计，但有一些边界情况没处理好：

**做对了的**：
- 2 小时 idle 阈值合理——大多数用户不会放下 2 小时以上再回来
- 摘要格式结构化（主题/目标/事实/未解决/线索）——确保关键信息不丢
- 240K token 强制摘要——防止超长对话超出 LLM 上下文窗口

**还可以改进的**：
- 摘要是在"达到阈值"时才触发，而不是"预测到即将超限"时提前触发
- 冷恢复只保留最后 8 轮，但用户的"长期兴趣"可能需要更细粒度的追踪（如话题标签）
- Dreaming 引擎（M3）本质上是另一个摘要系统——它和会话摘要是什么关系？目前是独立运作的，没有互相引用

::: exercise
设计一个改进方案：让冷热分层的摘要和 Dreaming 引擎的记忆更新联动。比如——Dreaming 发现用户最近关心"Python 异步编程"，这个信息如何影响该用户的会话摘要？

--- perspective
依据：当前两个系统独立运作。会话摘要关注"这次对话讲了什么"，Dreaming 关注"用户长期是什么样的人"。跨会话的行为模式（如反复问某类问题）目前没有进入会话摘要。

联动思路：Dreaming 生成 MEMORY.md 时，可以同时生成一个 `user_topics.json`（话题标签 + 置信度）。冷恢复时，摘要 prompt 可以附带这些话题标签作为"你可能关心"的提示。这不是技术难题，是设计决定。
---
:::

## [m8-prompt] Prompt Engineering 的分层哲学

深小理的系统提示词不是一个大长文——它是分层组合的：

```
SOUL.md (我是谁)
  + SUAT.md (我在哪)
  + MEMORY.md (我经历过什么)
  + interaction_policy.md (我怎么跟人说话)
  + tool_guidelines.md (我怎么用工具)
  + skills manifest (我有哪些技能)
  + session summary (这次对话讲什么)
  + user memory (用户是谁)
```

每层有独立的维护者和变更频率：

| 层 | 谁维护 | 变更频率 |
|----|--------|---------|
| SOUL.md | 设计者 | 几乎不变 |
| SUAT.md | 校园信息维护者 | 招生季/新数据时更新 |
| MEMORY.md | Dreaming 引擎 (自动) | 每次 dreaming 后 |
| interaction_policy / tool_guidelines | 开发者 | 工具变更时 |
| skills manifest | SkillRegistry (自动扫描) | 添加/删除 skill 时 |
| session summary | Agent (自动生成) | 每次触发摘要时 |
| user memory | Agent (用户确认后) | 偶尔 |

这个分层的好处：**每一层可以独立演化，不需要一次改一大段 prompt。** 坏处：**层与层之间可能冲突**（比如 MEMORY.md 说"我是热情的"，但 interaction_policy 说"回答要简洁"——Agent 需要自己判断什么时候热情什么时候简洁）。

## [m8-final] 最后的反思：你学到了什么

课程到这里就结束了。但这个项目能教的东西远不止这些——因为深小理有一个独特的特征：**它是用 AI 辅助构建的。**

这意味着什么？

1. **代码质量不来自写代码的人有多聪明，来自架构决策**——AI 可以写大量代码，但不会替你决定"冷热分层的阈值应该是 2 小时还是 4 小时"。

2. **系统提示词是新的"源代码"**——prompt 文件（SOUL.md / interaction_policy.md 等）和行为逻辑一样重要。改了 prompt 就是改了产品行为。

3. **好的架构让 AI 更好用**——清晰的模块边界（model_clients.py 统一入口、MemorySystem 单一 I/O 抽象、纯函数 reducer）让 AI 在改代码时不容易出错。

4. **安全不能靠 AI 自觉**——路径防护、privacy guard、rate limit 这些要在代码层卡死，不能指望 prompt 里的"你要注意安全"。
