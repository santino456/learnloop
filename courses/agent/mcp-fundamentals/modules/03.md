## [m3-scenario] 场景：CampusSchedule

假设你要给一个 AI 助手接入校园日程系统。系统里有三类能力：

- 读取某一天的课程表。
- 查询某个教室的预约状态。
- 生成一段“今天该怎么安排学习”的建议模板。

这个练习不要求你真的写 SDK 代码。目标是训练一个更基本的能力：**把需求分配到正确的 MCP 原语**。

::: checkpoint
如果只是把“课程表 schema”给模型看，这是 tool 还是 resource？

--- answer
更像 resource。它是上下文数据，不是让模型执行的动作。
---
:::

## [m3-pick-primitives] 先分清读、做、模板

把 CampusSchedule 拆成这三类：

| 需求 | 更合适的原语 | 理由 |
|------|--------------|------|
| 读取课程表 schema | Resource | 给模型理解数据结构 |
| 查询某天课程 | Tool | 需要带参数调用外部系统 |
| 查询教室状态 | Tool | 需要执行查询动作 |
| 生成学习安排建议 | Prompt | 这是用户可选择的工作流模板 |

::: exercise
下面哪个设计更合理？

- A. 把 `get_schedule`、`get_room_status`、`daily_study_plan_prompt` 都做成 tools。
- B. 把 schema 做成 resource，把两个查询做成 tools，把学习安排建议做成 prompt。
- C. 把所有内容都写进 server description，让模型自己猜。

--- answer
B
--- explanation
依据：MCP 官方文档把 resources、tools、prompts 分成不同原语。schema 是上下文，查询是动作，学习安排是可复用交互模板。
---
:::

## [m3-tool-shape] Tool 的 shape 要让模型不容易误用

一个好的 tool 名称应该表达动作和对象。比如：

```json
{
  "name": "campus_schedule_get_day",
  "title": "Get Daily Campus Schedule",
  "description": "Return classes and events for one date in the campus schedule system.",
  "inputSchema": {
    "type": "object",
    "properties": {
      "date": {
        "type": "string",
        "description": "Date in YYYY-MM-DD format"
      }
    },
    "required": ["date"]
  }
}
```

这不是官方示例，而是基于官方 tools 字段整理出的教学样例。真正实现时要按所用 SDK 的 API 写。

::: exercise
改写这个糟糕工具名，让它更适合模型选择：

```json
{
  "name": "query",
  "description": "query things"
}
```

--- answer
可以改成 `campus_room_get_availability`，description 写清楚它查询某个教室在某段时间是否可用。
--- explanation
好名字应该包含领域对象和动作，description 应说明参数、结果和何时使用。
---
:::

## [m3-resource-shape] Resource 不应该偷偷做动作

Resource 的价值是让 host 或模型拿到上下文。比如：

```json
{
  "uri": "campus://schedule/schema",
  "name": "schedule_schema",
  "title": "Campus Schedule Schema",
  "description": "Fields and meanings for the campus schedule database",
  "mimeType": "application/json"
}
```

如果读取这个 URI 会触发写入、发送通知、改变预约状态，那它就不再是一个干净的 resource。

::: checkpoint
为什么“读取 resource 时顺便刷新远程缓存”是危险信号？

--- answer
因为 resource 应该主要提供上下文数据。如果读取动作隐藏副作用，host 和用户就很难理解当前操作的真实风险。
---
:::

## [m3-prompt-shape] Prompt 是把成熟工作流交给用户选择

一个 prompt 可以是：

```json
{
  "name": "daily_study_plan",
  "title": "Plan Today’s Study",
  "description": "Use today’s campus schedule and user goals to draft a study plan.",
  "arguments": [
    {
      "name": "goal",
      "description": "What the learner wants to focus on today",
      "required": true
    }
  ]
}
```

Prompt 的好处是它保留用户主动性：用户知道自己选择了一个工作流，而不是模型在背后随便套模板。

## [m3-review] 自查：你的 server 是否过度暴露

::: decision
CampusSchedule 要不要提供 `campus_sql_query`，允许模型传任意 SQL？

- A. 可以，模型更灵活。
- B. 不建议，先提供边界清楚的查询 tools。
- C. 可以，但只给只读数据库。

--- perspective
依据：基于本章对 tool 边界、副作用和用户可理解性的练习；实际系统还要结合权限、审计和 SQL sandbox 验证。

B 是更好的默认判断。任意 SQL 把太多策略推给模型，用户也很难理解一次调用的真实范围。只读数据库能降低写入风险，但仍可能带来隐私、性能和越权查询问题。
---
:::
