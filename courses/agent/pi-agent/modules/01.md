## [m1-confusion] 一个值得先问的问题

市面上的 AI coding agent 已经很多：Claude Code、OpenAI Codex、Cursor、Windsurf 等。它们通常给你一个聊天窗口，让你用自然语言驱动代码修改。但如果你想要：

- 把 agent 嵌入自己的工具链；
- 自定义模型路由、认证或上下文压缩策略；
- 让 agent 遵守项目特定的安全规则；
- 把自己的内部系统包装成 agent 可调用的能力；

这些闭源或高度集成的产品往往只能在其既定框架内使用。

Pi 的定位是 **minimal terminal coding harness**，核心区别是：它不是让你适应某个固定 workflow，而是让你用 TypeScript 扩展它，甚至把它的底层包拆出来嵌入自己的应用。

> 来源：[Pi README — Philosophy](https://github.com/earendil-works/pi/blob/371adcf37130629ffb9bbeed9f5548ce08ffa93b/packages/coding-agent/README.md#philosophy)。

::: checkpoint
Pi 与 Claude Code / Codex 最显著的设计差异是什么？

A. Pi 的模型能力更强  
B. Pi 默认支持更多编程语言  
C. Pi 把可扩展性作为一等公民，允许用 TypeScript 扩展、嵌入或二次开发  
D. Pi 是一个闭源 SaaS 产品

--- answer
C。Pi 强调 adapt pi to your workflows，而不是反过来；其架构拆分为独立包，并支持 extensions、skills、prompt templates、themes 和 pi packages。
---
:::

## [m1-architecture] 四层架构

Pi 的仓库是一个 TypeScript monorepo，包含四个核心包：

| 包 | 职责 | 二次开发价值 |
|---|---|---|
| `@earendil-works/pi-ai` | 统一多厂商 LLM API | 如果你想自己搭一个支持 OpenAI/Anthropic/Google 等 provider 的 agent，这是底层引擎 |
| `@earendil-works/pi-agent-core` | 状态化 agent loop + 事件流 | 你可以拿它构建自己的 agent 运行时，而不依赖 Pi CLI |
| `@earendil-works/pi-coding-agent` | 终端 CLI + TUI + sessions + 扩展机制 | 直接面向用户的 coding harness，支持扩展 |
| `@earendil-works/pi-tui` | 差分渲染的终端 UI 库 | 如果你要做一个复杂 TUI，可以直接复用 |

> 来源：[Pi README — All Packages](https://github.com/earendil-works/pi/blob/371adcf37130629ffb9bbeed9f5548ce08ffa93b/README.md#all-packages)。

这种分层意味着：你可以只取 `pi-ai` 和 `pi-agent-core` 做一个后台 agent，也可以基于 `pi-coding-agent` 做一个带扩展的终端产品，还可以只复用 `pi-tui` 做界面。

::: checkpoint
Pi 的四个核心包中，负责“状态化 agent loop 和事件流”的是哪一个？

--- answer
`@earendil-works/pi-agent-core`。
---
:::

## [m1-philosophy] 设计哲学：少即是多，扩展来补

Pi  README 里有一段关键描述：

> "Pi ships with powerful defaults but skips features like sub agents and plan mode. Instead, you can ask pi to build what you want or install a third party pi package that matches your workflow."

也就是说，Pi 的核心团队不试图在 CLI 里内置所有高级功能（子 agent、plan mode 等），而是提供可扩展点：

- **Extensions**：用 TypeScript 改行为、加 tool、加命令、加 UI；
- **Skills**：按 [Agent Skills](https://agentskills.io/specification) 标准封装能力；
- **Prompt Templates**：复用提示词模板；
- **Themes**：自定义界面主题；
- **Pi Packages**：通过 npm 或 git 分发上述能力。

> 来源：[Pi README — Pi is a minimal terminal coding harness](https://github.com/earendil-works/pi/blob/371adcf37130629ffb9bbeed9f5548ce08ffa93b/packages/coding-agent/README.md)。

::: checkpoint
Pi 为什么不内置 sub agents 和 plan mode？

A. 因为实现起来太难  
B. 因为用户不需要  
C. 因为 Pi 选择用扩展机制让用户按需安装，而不是把功能硬编码进 CLI  
D. 因为模型不支持

--- answer
C。Pi 的哲学是可扩展性优先：核心保持最小，高级能力通过 extensions、skills 或第三方 pi packages 引入。
---
:::

## [m1-why-extend] 为什么值得二次开发

从开发者角度看，Pi 值得投入的原因包括：

1. **可组合**：底层包可以单独使用，不一定要用完整 CLI。
2. **可扩展**： extensions 能拦截事件、注册 tool、替换 UI，几乎能改造任意行为。
3. **可分发**：skills 和 pi packages 让个人或团队的工作流可以复用和共享。
4. **可审计**：MIT 协议、源代码开放，供应链加固策略写得非常明确。
5. **可嵌入**：提供 RPC、JSON、SDK 模式，方便接入其他应用（如 [openclaw/openclaw](https://github.com/openclaw/openclaw)）。

> 来源：[Pi README — Modes](https://github.com/earendil-works/pi/blob/371adcf37130629ffb9bbeed9f5548ce08ffa93b/packages/coding-agent/README.md#programmatic-usage) 与 [AGENTS.md — Releasing](https://github.com/earendil-works/pi/blob/371adcf37130629ffb9bbeed9f5548ce08ffa93b/AGENTS.md)。

::: checkpoint
以下哪项不是 Pi 适合二次开发的理由？

A. 可组合的底层包  
B. 闭源但 API 文档完善  
C. 可扩展的 extension/skill 机制  
D. 支持 SDK/RPC/JSON 等多种嵌入方式

--- answer

---
:::

## [m1-summary] 模块小结

- Pi 是一个分层的 AI agent 工具链，核心包为 `pi-ai`、`pi-agent-core`、`pi-coding-agent`、`pi-tui`。
- 它的设计哲学是“最小核心 + 强大扩展”，而不是把所有功能都内置。
- 对开发者来说，最有价值的是：可以单独使用底层包、写 extensions、封装 skills、分发 pi packages。
- 后续模块会逐层拆开这四个包，并动手写一个 extension 和一个 skill。
