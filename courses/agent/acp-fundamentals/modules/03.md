## [m3-handshake] initialize 握手

`initialize` 是客户端和 Agent 之间的第一次结构化对话。它的任务是确认协议兼容，并互相通报各自支持的能力。

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "initialize",
  "params": {
    "capabilities": {},
    "clientInfo": {"name": "minimal-client", "version": "0.1.0"}
  }
}
```

> 验证提示：具体字段名和协议版本可能变化。在把示例写成可运行代码前，先对照你使用的 ACP 实现或 CLI 帮助文档确认。

::: checkpoint
initialize 请求里，哪一部分用来告诉 Agent“我是谁”？

--- answer
`clientInfo` 对象，包含 `name` 和 `version`。
---
:::

## [m3-capabilities] 能力协商

能力协商（capabilities）让双方不必互相猜测。客户端可以说自己支持哪些资源、工具或交互模式；Agent 也可以说它能处理什么。

重要的不是背下某一张固定的能力矩阵，而是养成习惯：**在使用可选功能之前，先检查双方都支持什么**。

## [m3-failure] 协商失败怎么办

如果握手失败，就不要继续假装会话有效。应该抛出明确错误，说明是哪一项必需能力或协议版本没有匹配上。

::: checkpoint
假设 Agent 要求 `streaming` 能力，但客户端没声明支持，接下来应该怎么处理？

--- answer
停止会话并报告：缺少必需的 `streaming` 能力。继续发送后续请求可能导致未定义行为。
---
:::

## [m3-summary] 模块小结

- initialize 在客户端和 Agent 之间建立了第一份可靠契约。
- 能力协商让可选功能变得可控。
- 具体协议字段必须针对当前实现版本验证。

::: exercise
initialize 请求中的哪两个字段让双方互相了解身份与支持的功能？

- A. `method` 和 `id`
- B. `params` 和 `jsonrpc`
- C. `clientInfo` 和 `capabilities`
- D. `stdout` 和 `stderr`

--- answer
C
--- explanation
`clientInfo` 说明客户端是谁，`capabilities` 说明各自支持什么。`method`/`id`/`params` 是 JSON-RPC 通用字段，`stdout`/`stderr` 是本地传输通道。
:::

::: exercise
补全：客户端向 Agent 的 ________ 写入 JSON，Agent 向 ________ 返回 JSON，日志走 ________。

--- answer
stdin; stdout; stderr
--- explanation
行分隔 JSON 的本地模式：stdin 进，stdout 出，stderr 留给日志。
:::
