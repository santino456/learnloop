## [m3-why] 为什么需要 pi-agent-core

`pi-ai` 已经能让我们发起一次 LLM 调用并处理流式事件。但一个 coding agent 真正复杂的是：

- 维护跨多轮对话的状态；
- 处理模型触发的工具调用；
- 把工具结果重新喂给模型；
- 支持用户中途插入 steering/follow-up 消息；
- 允许 UI 监听事件并更新界面；
- 处理错误、重试、上下文压缩。

`pi-agent-core` 把这些封装成一个 **状态化 agent runtime**，并且仍然保持与 UI 解耦：它只发射事件，不假设前端是什么样子。

> 来源：[packages/agent/README.md — @earendil-works/pi-agent-core](https://github.com/earendil-works/pi/blob/371adcf37130629ffb9bbeed9f5548ce08ffa93b/packages/agent/README.md)。

::: checkpoint
pi-agent-core 的主要职责是什么？

A. 渲染终端 UI  
B. 管理多轮状态、工具调用循环和事件流  
C. 下载 npm 包  
D. 提供多 provider LLM API

--- answer
B。pi-agent-core 是状态化 agent runtime，负责多轮对话、工具执行和事件发射。
---

## [m3-message-types] AgentMessage vs LLM Message

这是 pi-agent-core 最重要的设计之一：

- **AgentMessage**：agent 内部使用的消息类型，可以包含 LLM 无法理解的 UI 消息、自定义消息类型。
- **Message**：真正发给 LLM 的消息，只有 `user`、`assistant`、`toolResult` 三种角色。

转换关系：

```
AgentMessage[] → transformContext() → AgentMessage[] → convertToLlm() → Message[] → LLM
```

`transformContext` 用于剪枝、压缩、注入外部上下文；`convertToLlm` 用于过滤掉 UI-only 消息。

> 来源：[packages/agent/README.md — Core Concepts](https://github.com/earendil-works/pi/blob/371adcf37130629ffb9bbeed9f5548ce08ffa93b/packages/agent/README.md#core-concepts)。

::: checkpoint
AgentMessage 与 LLM Message 分离的好处是什么？

--- answer
可以在 agent 内部保留 UI 消息、自定义消息类型，而不污染真正发给 LLM 的上下文；同时支持 context pruning 和 compaction。
---
:::

## [m3-event-flow] prompt() 事件序列

调用 `agent.prompt("Hello!")` 时，事件按严格顺序发射：

```
agent_start
├─ turn_start
│  ├─ message_start (user)
│  ├─ message_end   (user)
│  ├─ message_start (assistant)
│  ├─ message_update (text_delta ...)
│  ├─ message_end   (assistant)
│  └─ turn_end
└─ agent_end
```

如果模型调用了工具，流程会继续：

```
... assistant message with toolCall ...
├─ tool_execution_start
├─ tool_execution_update (可选，工具流式输出)
├─ tool_execution_end
├─ message_start/end (toolResult)
├─ turn_end
│
├─ turn_start
├─ message_start (assistant response to tool result)
...
```

> 来源：[packages/agent/README.md — Event Flow](https://github.com/earendil-works/pi/blob/371adcf37130629ffb9bbeed9f5548ce08ffa93b/packages/agent/README.md#event-flow)。

::: checkpoint
`turn_start` 和 `turn_end` 之间一定包含哪些事件？

--- answer
至少包含 user 和 assistant 的 `message_start`/`message_end`，以及可能的 `message_update` 和工具执行事件。
---
:::

## [m3-loop-impl] agent-loop.ts 的实现骨架

核心循环在 `packages/agent/src/agent-loop.ts` 中实现。它的主要结构是：

```typescript
export function agentLoop(
  prompts: AgentMessage[],
  context: AgentContext,
  config: AgentLoopConfig,
  signal?: AbortSignal,
  streamFn?: StreamFn,
): EventStream<AgentEvent, AgentMessage[]> { ... }
```

`agentLoop` 是一个低层函数，返回 `EventStream`。它内部启动 `runAgentLoop`，后者会：

1. 把 prompts 加入 context；
2. 发射 `agent_start`、`turn_start`、user message 事件；
3. 进入 `runLoop`；
4. 在循环中处理 steering 消息、流式 assistant 响应、工具调用、重试等；
5. 最终发射 `agent_end` 并返回新增的消息列表。

> 来源：[packages/agent/src/agent-loop.ts](https://github.com/earendil-works/pi/blob/371adcf37130629ffb9bbeed9f5548ce08ffa93b/packages/agent/src/agent-loop.ts#L31-L54)。

::: checkpoint
`agentLoop()` 返回什么类型？

--- answer
`EventStream<AgentEvent, AgentMessage[]>`。
---
:::

## [m3-tool-execution] Tool 执行模式

pi-agent-core 支持两种工具执行模式：

- `parallel`（默认）：预检（preflight）工具调用顺序执行，允许的 tool 并发执行，完成后按 assistant 原顺序放回 toolResult。
- `sequential`：按顺序逐个执行。

模式可以在全局 `toolExecution` 配置，也可以对每个 tool 设置 `executionMode`。如果同一批调用中有任何 tool 标记为 `sequential`，整批都会退化为顺序执行。

> 来源：[packages/agent/README.md — Tool Execution Mode](https://github.com/earendil-works/pi/blob/371adcf37130629ffb9bbeed9f5548ce08ffa93b/packages/agent/README.md#with-tool-calls)。

::: checkpoint
为什么某一批工具调用会整体退化为 sequential？

--- answer
只要该批次中有一个 tool 显式设置 `executionMode: "sequential"`，pi-agent-core 就会为了保证正确性而整批顺序执行。
---
:::

## [m3-hooks] 生命周期 Hooks

pi-agent-core 提供多个 hook 让扩展代码介入循环：

| Hook | 触发时机 | 用途 |
|---|---|---|
| `transformContext` | 每次发给 LLM 前 | 剪枝、压缩、注入外部上下文 |
| `convertToLlm` | `transformContext` 之后 | 过滤自定义消息 |
| `beforeToolCall` | 参数校验后、执行前 | 权限门、确认框、阻断危险操作 |
| `afterToolCall` | 执行完成后 | 日志、审计、后处理 |
| `prepareNextTurn` | turn 之间 | 动态调整 model/tools/config |
| `shouldStopAfterTurn` | turn 结束后 | 决定是否继续循环 |

> 来源：[packages/agent/README.md — Agent Options](https://github.com/earendil-works/pi/blob/371adcf37130629ffb9bbeed9f5548ce08ffa93b/packages/agent/README.md#agent-options) 与 [packages/agent/src/agent.ts](https://github.com/earendil-works/pi/blob/371adcf37130629ffb9bbeed9f5548ce08ffa93b/packages/agent/src/agent.ts#L95-L116)。

::: checkpoint
`beforeToolCall` 可以做什么？

--- answer
在工具执行前阻断调用、弹出确认框、记录日志、做权限检查等。
---
:::

## [m3-queues] Steering 与 Follow-up 队列

`Agent` 类维护两个内部队列：

- **steeringQueue**：用户在 agent 工作时发送的消息，在当前 turn 的工具执行结束后插入。
- **followUpQueue**：用户在 agent 工作时发送的消息，等 agent 完全空闲后才插入。

两种队列都支持 `one-at-a-time` 和 `all` 两种出队模式，对应 Pi CLI 中的 `/settings` 配置。

> 来源：[packages/agent/src/agent.ts](https://github.com/earendil-works/pi/blob/371adcf37130629ffb9bbeed9f5548ce08ffa93b/packages/agent/src/agent.ts#L118-L152) 与 [packages/coding-agent/README.md — Message Queue](https://github.com/earendil-works/pi/blob/371adcf37130629ffb9bbeed9f5548ce08ffa93b/packages/coding-agent/README.md#message-queue)。

::: checkpoint
steering 消息与 follow-up 消息的关键区别是什么？

--- answer
steering 在当前 turn 的工具执行结束后立即生效；follow-up 等 agent 完全空闲后才生效。
---
:::

## [m3-summary] 模块小结

- `pi-agent-core` 在 `pi-ai` 之上封装了状态化多轮 agent loop。
- `AgentMessage` 与 LLM `Message` 分离，支持自定义消息、context pruning 和 compaction。
- 事件流严格有序：`agent_start` → `turn_start` → messages/tool executions → `turn_end` → `agent_end`。
- 工具执行支持 `parallel`/`sequential`，并可通过 hooks 实现权限控制、日志、动态配置。
- `Agent` 类提供 steering 和 follow-up 队列，允许用户在 agent 运行时插入消息。

> 来源：本地 m2《pi-ai：统一 LLM API》；外部 [packages/agent/README.md](https://github.com/earendil-works/pi/blob/371adcf37130629ffb9bbeed9f5548ce08ffa93b/packages/agent/README.md)。
