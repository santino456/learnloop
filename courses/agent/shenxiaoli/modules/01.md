## [m1-what] 深小理是什么

深小理是深圳理工大学的 AI 智能伙伴——一个基于 LangGraph + React 的全栈 Web 应用。

从学习架构的角度看，它不是“一个聊天网页”这么简单。它至少包含五层：

- 浏览器里的聊天界面。
- 负责转发请求和静态资源的 Web 入口。
- 负责登录、会话、限流和流式响应的后端 API。
- 负责推理、工具调用和记忆读写的 Agent。
- 负责持久化的数据库和对象存储。

这门课第一章只做一件事：把这些层放到一张图里，让你先知道一次用户提问会经过哪些系统。具体字段、节点数量、部署细节需要以当前代码和配置为准；没有来源支持的项目历史和性能数字不在本章里写成事实。

## [m1-stack] 技术栈全景

```
浏览器 ──→ Nginx ──→ FastAPI (Python) ──→ LangGraph Agent ──→ LLM API
   │                     │                       │
   └── OSS (图片)        ├── PostgreSQL 14       ├── 记忆系统 (session + 持久)
                         ├── JWT 认证            ├── 工具注册 (7 tools)
                         ├── 速率限制            ├── 技能系统 (5 skills)
                         └── 管理后台            └── Dreaming 引擎
```

| 层 | 技术 | 为什么选它 |
|----|------|-----------|
| 前端 | React 18 + TypeScript + Vite + Tailwind | 标准选型，Vite 构建快；Tailwind 零配置 |
| 后端 | FastAPI + Python 3.11 | async 原生支持，对 SSE 流式友好；LangGraph 生态在 Python |
| Agent 框架 | LangGraph StateGraph | 有向图建模 ReAct 循环，支持中断/恢复（privacy confirm） |
| 数据库 | PostgreSQL 14 + psycopg 3 | 事务可靠；裸机部署最简单；JSONB 存 messages 够用 |
| LLM | OpenAI 兼容 API | 不绑定单一模型供应商 |
| 图片 | 阿里云 OSS | 图片走对象存储，避免把大文件都压到应用后端 |
| 部署 | Nginx + systemd | 裸机部署，一条 `deploy.sh` 搞定 |

> 如果你对上面某些技术还不熟悉，可以先跳到 [m9-技术栈速查](../m9.html) 查概念，再回来继续。

## [m1-division] 全栈职责划分

如果你对前后端架构不熟，可以先用一句话理解：

**前端负责让用户看见和操作，后端负责让系统可信地执行。**

在深小理里，前端主要做这些事：

- 显示聊天 UI、会话列表和富文本内容。
- 把用户输入发给后端。
- 消费后端通过 SSE 推来的增量消息。
- 维护浏览器里的登录态和页面交互状态。

后端主要做这些事：

- 校验用户身份、检查速率限制、读写会话。
- 调用 Agent 图，让模型决定下一步是回复、调用工具还是继续推理。
- 执行工具、读取记忆、把结果写回数据库。
- 用 SSE 把 Agent 的中间过程和最终回答推给前端。

所以一次普通提问可以这样理解：

```text
用户输入
  → React 收集文本
  → FastAPI 创建/读取会话
  → LangGraph Agent 进入推理循环
  → 工具和记忆系统提供上下文
  → LLM 生成增量回复
  → SSE 把内容流回浏览器
  → 前端逐步渲染出来
```

这个边界很重要：前端不应该自己决定 Agent 怎么思考；后端也不应该关心按钮、动画和 Markdown 怎么显示。

## [m1-baremetal] 为什么裸机部署而不是 Docker

这一节不是说裸机部署一定比 Docker 好，而是训练你看懂部署取舍。对于一个还在快速迭代的单体 Web 应用，Nginx + systemd 的裸机部署有几个直接好处：

**正面收益**：
- **路径短**：`deploy.sh` 可以直接完成拉代码、装依赖、构建前端、重启服务。
- **排查直观**：`journalctl -u shenxiaoli -f` 看日志，`systemctl status` 看进程状态。
- **组件少**：没有镜像、容器网络、volume、compose 文件这些额外层。
- **适合单体**：如果只有一个后端服务和一个前端构建产物，裸机部署的认知负担更低。

**代价和边界**：
- 没有容器化的环境隔离——依赖 Python venv + npm 管理。
- 没有多容器编排——服务一多，依赖关系会变难管理。
- 部署一致性依赖服务器环境——换机器时需要重新整理系统依赖。
- 滚动发布、灰度发布、快速回滚这些能力需要额外设计。

::: checkpoint
列出深小理选裸机部署的三个正面理由和两个代价。然后想想：如果你的应用有 10 个微服务，这些理由还成立吗？

--- answer
正面：1) 部署路径短；2) journalctl/systemctl 排查直观；3) 组件少，适合单体应用。
代价：1) 环境隔离弱，依赖 venv/npm 和服务器状态；2) 服务数量变多后难以管理依赖和部署顺序。
如果 10 个微服务，裸机管理会变得难以追踪依赖和部署顺序，Docker Compose 或 K8s 才有意义。裸机适合单机单体应用，不适合多服务编排。
--- 
:::

## [m1-agents] 项目契约：AGENTS.md

深小理有一个有意思的设计模式：**项目文档即架构契约**。

`AGENTS.md` 是公开的（进入 Git）、`AGENTS_LOCAL.md` 是私有的（gitignore）。两者共同构成"新 Agent 或新开发者进入项目时需要知道的一切"：

- `AGENTS.md` 写：技术栈、项目结构、本地开发流程、部署流程、架构要点、模型配置规范。
- `AGENTS_LOCAL.md` 写：服务器 IP、SSH 密钥路径、数据库密码、OSS bucket 名称。

::: checkpoint
看看你自己的项目，有没有一个类似 AGENTS.md 的入口文档？如果没有，试着写一个——就写 10 行：技术栈、目录结构、怎么跑起来、部署怎么走。

--- answer
好的项目文档不是为了"规范"，是为了减少重复沟通成本。没有 AGENTS.md 的时候，每次让 AI 理解项目需要重新解释；有 AGENTS.md 后，一次写好，所有新 Session 自动注入。这在 AI 时代尤其重要：AGENTS.md 是写给人 + AI 共用的项目上下文，不只是给人看的 README。
---
:::

## [m1-summary] 模块小结

- 深小理可以先理解成五层：浏览器、Web 入口、后端 API、Agent、持久化存储。
- 前端负责展示和交互，后端负责身份、会话、Agent 循环、工具执行和数据写入。
- SSE 的价值是让后端把 Agent 的增量输出推给浏览器，避免用户只能等最终结果。
- 裸机部署适合快速迭代的单体应用，但环境隔离、回滚和多服务编排都需要额外考虑。
- `AGENTS.md` 是人 + AI 共用的项目契约——一次写好，长期复用。
