## [m2-attention] 自注意力快速回顾

Transformer 的自注意力计算可以概括为：

```text
Q = X · W_Q
K = X · W_K
V = X · W_V

Attention(Q, K, V) = softmax(Q · K^T / sqrt(d_k)) · V
```

其中 X 是输入序列的隐藏状态，W_Q、W_K、W_V 是可学习权重。Q、K、V 分别是查询、键、值矩阵。

关键点：**每个 token 的 K 和 V 只依赖于它自己的隐藏状态和权重矩阵**，不依赖后面的 token。这就是 KV Cache 可行的数学基础。

> 为什么 K 和 V 可以在生成后就被缓存？因为每个 token 的 K 和 V 只依赖它自己的隐藏状态 X_i 和权重矩阵 W_K、W_V，不依赖序列中后面的 token。这个因果性使得 K、V 可以在生成后立即缓存复用。

## [m2-step-by-step] 逐 token 生成过程

假设序列已有 3 个 token，现在要生成第 4 个：

**没有 KV Cache 时：**

1. 把全部 4 个 token 的隐藏状态拼成 X（4 行）。
2. 重新计算 Q、K、V（都是 4 行）。
3. 计算 4×4 的注意力矩阵。
4. 得到输出。

**有 KV Cache 时：**

1. 只计算第 4 个 token 的隐藏状态 X_4。
2. 用 X_4 计算 Q_4、K_4、V_4（各 1 行）。
3. 把 K_4 追加到缓存的 K 中（缓存从 3 行变成 4 行）。
4. 把 V_4 追加到缓存的 V 中。
5. 用 Q_4 和缓存中全部 4 行 K 计算注意力分数。
6. 用注意力分数对全部 4 行 V 加权求和。

| 对比项 | 无 KV Cache | 有 KV Cache |
|--------|-------------|-------------|
| 每步前向传播的 token 数 | 全部 N 个 | 仅 1 个新 token |
| K、V 计算 | 全部重算 | 只算新 token 的 |
| 注意力矩阵 | N×N | 1×N |
| 显存 | 不额外存 K、V | 存全部历史 K、V |

> 有 KV Cache 时，生成第 4 个 token 需要计算的注意力矩阵是 1×4：只需要新 token 的 Q（1 行）和缓存中全部 4 个 token 的 K（4 行）做点积。

## [m2-multi-layer] 多层 Transformer 的缓存

Transformer 有 L 层，每层都有独立的注意力机制。因此 KV Cache 也要按层存储：

```text
Layer 0:  K_cache[0] = [k_0, k_1, ..., k_n]  V_cache[0] = [v_0, v_1, ..., v_n]
Layer 1:  K_cache[1] = [k_0, k_1, ..., k_n]  V_cache[1] = [v_0, v_1, ..., v_n]
...
Layer L-1: K_cache[L-1] = [...]               V_cache[L-1] = [...]
```

每一层的 KV Cache 在该层的前向传播中使用，不跨层共享。第 l 层的输出是第 l+1 层的输入。

> 注意：前馈网络（FFN）层不需要缓存，因为 FFN 对每个 token 独立计算，没有跨 token 依赖。

## [m2-multi-head] 多头注意力的缓存

多头注意力下，每层有 H 个头，每个头有独立的 K 和 V。因此每层的 KV Cache 实际上是：

```text
K_cache[layer][head] = [k_0, k_1, ..., k_n]   维度: (seq_len, d_head)
V_cache[layer][head] = [v_0, v_1, ..., v_n]   维度: (seq_len, d_head)
```

总缓存大小 = 2（K 和 V）× L（层数）× H（头数）× seq_len × d_head × batch_size。

## [m2-append] 追加操作

KV Cache 的核心操作是**追加**：每生成一个新 token，把它的 K 和 V 追加到缓存末尾。

```python
# 伪代码：KV Cache 追加操作
new_k = compute_k(new_token_hidden)   # (1, d_head)
new_v = compute_v(new_token_hidden)   # (1, d_head)

k_cache = torch.cat([k_cache, new_k], dim=0)  # (seq_len+1, d_head)
v_cache = torch.cat([v_cache, new_v], dim=0)  # (seq_len+1, d_head)

# 注意力计算
attn_scores = new_q @ k_cache.T / sqrt(d_head)  # (1, seq_len+1)
attn_weights = softmax(attn_scores)
output = attn_weights @ v_cache                  # (1, d_head)
```

在实际实现中，通常会预分配最大长度的缓存空间，然后用一个指针标记当前有效长度，避免频繁的内存分配。

> 为什么预分配而不是动态扩展？避免频繁的内存分配和拷贝操作（如 torch.cat 会创建新张量）。预分配后只需移动指针，性能更好。代价是固定了最大序列长度。

## [m2-prefill] Prefill 阶段

在生成第一个新 token 之前，模型需要处理整个输入 prompt。这个阶段叫 **prefill**（预填充）：

1. 把 prompt 的全部 token 一次性前向传播。
2. 计算每一层每个头的 K 和 V。
3. 把它们存入 KV Cache。

Prefill 之后，KV Cache 就包含了 prompt 的全部历史信息，后续的 decode 阶段每步只处理 1 个 token。

> Prefill 是计算密集的（处理 N 个 token），decode 是访存密集的（每步处理 1 个 token 但要读取全部 KV Cache）。这个区分对后续的优化策略很重要。

## [m2-summary] 模块小结

- 每个 token 的 K、V 只依赖自身隐藏状态，这是 KV Cache 可行的数学基础。
- 生成新 token 时只需计算它的 Q、K、V，然后追加 K、V 到缓存。
- 多层、多头各自独立缓存，不跨层跨头共享。
- Prefill 阶段一次性填充 prompt 的 KV Cache，decode 阶段逐步追加。
- Prefill 计算密集，decode 访存密集。

::: checkpoint
Prefill 和 decode 阶段的主要区别是什么？

--- answer
Prefill 是一次性处理整个 prompt 并填充 KV Cache，计算密集；decode 是逐 token 生成，每步只处理 1 个新 token 但读取全部缓存，访存密集。
