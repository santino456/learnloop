## [m3-formula] 显存占用公式

KV Cache 的显存占用可以用一个公式概括：

```text
KV Cache 显存 = 2 × L × H × d_head × seq_len × batch_size × dtype_size
```

各项含义：

| 符号 | 含义 | 典型值 |
|------|------|--------|
| 2 | K 和 V 两个矩阵 | 固定 |
| L | Transformer 层数 | 32 ~ 80 |
| H | 注意力头数 | 32 ~ 64 |
| d_head | 每个头的维度 | 64 ~ 128 |
| seq_len | 序列长度 | 2048 ~ 128K |
| batch_size | 并发请求数 | 1 ~ 256 |
| dtype_size | 每个元素的字节数 | 2（FP16/BF16） |

::: checkpoint
KV Cache 显存公式中的 "2" 代表什么？

--- answer
代表 K 和 V 两个矩阵都需要缓存。每个 token 在每层每个头都有一份 K 和一份 V。
---
:::

## [m3-example] 实际计算

以 Llama-2-7B 为例：

| 参数 | 值 |
|------|-----|
| L（层数） | 32 |
| H（头数） | 32 |
| d_head（头维度） | 128 |
| 模型权重显存（FP16） | ~14 GB |

计算 batch_size=1、seq_len=4096、FP16 下的 KV Cache 显存：

```text
= 2 × 32 × 32 × 128 × 4096 × 1 × 2 字节
= 2 × 32 × 32 × 128 × 4096 × 2
= 2,147,483,648 字节
= 2 GB
```

单请求 2 GB。如果 batch_size=32，就是 64 GB——远超模型权重。

::: checkpoint
同样是 Llama-2-7B、seq_len=4096、FP16，batch_size 从 1 增加到 32，KV Cache 显存从 2 GB 变成多少？

--- answer
64 GB。KV Cache 显存和 batch_size 成正比，32 × 2 GB = 64 GB。
---
:::

## [m3-vs-weights] 和模型权重的对比

| 模型 | 权重（FP16） | KV Cache/请求（seq=4096） | batch 何时超过权重 |
|------|-------------|--------------------------|-------------------|
| 7B | ~14 GB | ~2 GB | batch ≈ 7 |
| 13B | ~26 GB | ~3 GB | batch ≈ 9 |
| 70B | ~140 GB | ~10 GB | batch ≈ 14 |

> 关键观察：KV Cache 的显存占用随 batch size 线性增长。在多并发场景下，KV Cache 往往成为显存瓶颈，而非模型权重。

## [m3-compute] 计算节省

没有 KV Cache 时，生成第 N 个 token 的注意力计算量：

```text
Q·K^T: N × d × N = N² × d 次乘加
softmax·V: N × N × d = N² × d 次乘加
总计: 2 × N² × d
```

有 KV Cache 时：

```text
Q·K^T: 1 × d × N = N × d 次乘加
softmax·V: 1 × N × d = N × d 次乘加
总计: 2 × N × d
```

计算量从 O(N²) 降到 O(N)，但更重要的是省去了前 N-1 个 token 的全部前向传播（包括 FFN 等），这才是实际加速的主要来源。

::: checkpoint
有 KV Cache 时，生成第 N 个 token 的注意力计算量从 O(N² × d) 降到了什么？

--- answer
O(N × d)。只需要 1 个新 token 的 Q 和 N 个缓存 K 做点积，再对 N 个 V 加权求和。
---
:::

## [m3-throughput] 对吞吐的影响

KV Cache 对吞吐的影响有两面性：

- **正面**：减少了每步计算量，单 token 生成更快。
- **负面**：占用了更多显存，限制了能同时处理的 batch size。

当显存是瓶颈时（通常在长上下文或多并发场景），KV Cache 的显存占用直接决定了最大吞吐量。

::: checkpoint
KV Cache 对吞吐的负面影响是什么？

--- answer
KV Cache 占用显存，限制了能同时加载的 batch size。当显存成为瓶颈时，能并发的请求数减少，最大吞吐量受限。
---
:::

## [m3-summary] 模块小结

- KV Cache 显存 = 2 × L × H × d_head × seq_len × batch × dtype_size。
- 7B 模型单请求 seq=4096 的 KV Cache 约 2 GB，batch=32 时达 64 GB。
- KV Cache 显存随 batch 线性增长，多并发下常超过模型权重。
- 计算量从 O(N²) 降到 O(N)，但实际加速主要来自避免重复前向传播。
- KV Cache 是用显存换算力，显存限制反过来制约吞吐。

::: exercise
一个 32 层、32 头、头维度 128 的模型，batch_size=8，seq_len=2048，使用 FP16。KV Cache 占用多少显存？

--- answer
8 GB
--- explanation
计算：2 × 32 × 32 × 128 × 2048 × 8 × 2 = 8,589,934,592 字节 ≈ 8 GB。注意 batch_size=8 让显存比单请求翻了 8 倍。
:::

::: exercise
以下哪种情况会导致 KV Cache 显存占用最大？

- A. batch=1, seq_len=2048, 7B 模型
- B. batch=4, seq_len=4096, 7B 模型
- C. batch=1, seq_len=8192, 70B 模型
- D. batch=8, seq_len=4096, 7B 模型

--- answer
C
--- explanation
70B 模型层数和头数远多于 7B（L≈80, H≈64），即使 batch=1，seq_len=8192 下的 KV Cache 也约为 2 × 80 × 64 × 128 × 8192 × 1 × 2 ≈ 20 GB，远超其他选项。
:::
