## [m1-problem] 集成困境

想象一下：有三款编辑器，又有三个编码 Agent。如果没有共享协议，每款编辑器都要为每个 Agent 单独写适配器；每个 Agent 厂商也要去理解每款编辑器的私有集成接口。

这根本没法扩展。问题不在模型够不够聪明，而在**客户端和 Agent 之间的接口**没有统一标准。

::: concept
title: ACP 的核心心智模型
why: ACP 不是让 Agent 更聪明，而是把编辑器和 Agent 的集成关系从 N×M 降到 N+M。

把客户端想成“会发起会话和展示结果的界面”，把 Agent 想成“会执行推理和工具调用的进程”。协议的价值是让两边说同一种结构化语言。
:::

::: compare
left: 没有共享协议
right: 使用 ACP

- 集成方式 | 每个编辑器适配每个 Agent | 编辑器和 Agent 各实现标准接口
- 用户体验 | Agent 选择常被客户端绑定 | 用户更容易切换兼容 Agent
- 维护成本 | 新组合带来新适配器 | 新客户端或 Agent 可复用协议能力
:::

## [m1-solution] ACP 的回答

ACP 在客户端和 Agent 之间放了一份标准契约。客户端可以是编辑器、终端工具或本地应用；Agent 则是一个接收请求、申请工具调用、返回结果的进程。

> 核心思路：只要实现一次 ACP 客户端，或实现一次兼容 ACP 的 Agent，就能在多个说同一种协议的工具之间复用这套集成。

::: evidence
claim: ACP 标准化 code editors/IDEs 与 coding agents 之间的通信，并覆盖 local 和 remote 场景。
source: [Agent Client Protocol introduction](https://agentclientprotocol.com/get-started/introduction)
status: verified
basis: 官方介绍页明确描述 ACP 的目标和本地/远程适用范围。
:::

::: checkpoint
为什么“统一接口”比“让每个 Agent 更聪明”更能解决集成问题？

--- answer
因为集成痛苦主要来自 N×M 的适配器组合，而不是模型能力。统一接口把 N×M 变成 N+M。
---
:::

## [m1-roles] 谁受益

- **编辑器和 IDE 开发者**：只需要走一条客户端集成路径。
- **Agent 开发者**：只需要维护一份服务端协议接口。
- **用户**：客户端和 Agent 不再强耦合，选择变多了。

## [m1-modes] 本地与远程模式

最具体的模式是**本地 Agent 进程**：客户端把 Agent 作为子进程启动，通过标准输入输出交换行分隔的 JSON 消息。

远程模式则走网络传输。除非你能对照当前 ACP 文档确认细节，否则把远程实现当作“实现相关”来处理。

::: checkpoint
本地 ACP 最常用的传输形态是什么？

--- answer
行分隔的 JSON，通过 stdin/stdout 交换，日志走 stderr。
---
:::

## [m1-not] ACP 不是什么

ACP 不是 MCP 的替代品。MCP 解决的是 Agent 能使用哪些工具和数据源；ACP 解决的是**客户端如何驱动一次 Agent 会话**。

ACP 也不一定适合所有聊天机器人。如果你的应用只发一个 HTTP 请求、收一个回答，那普通 API 调用就够了。

## [m1-summary] 模块小结

- ACP 标准化了客户端到 Agent 的通信。
- 它减少了客户端与 Agent 之间的定制适配工作。
- 最适合本地、会用工具、有会话感的 Agent。
- 选择 ACP 的理由是它让集成变简单，而不是因为协议这个概念很时髦。
