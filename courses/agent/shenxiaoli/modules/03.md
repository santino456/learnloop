## [m3-two-memories] 两种记忆，两种生命周期

深小理有两套记忆系统，职责和生命周期完全不同：

| | Session 记忆 | 持久记忆 |
|---|---|---|
| **存储** | `conversations` 表的 messages（JSONB） | `users.user_md`（Markdown）+ `agent_observations` 表 |
| **作用** | 在单次对话中记住上下文 | 跨会话记住用户是谁、经历过什么 |
| **生命周期** | 随会话 | 随用户账号 |
| **谁在管** | Agent 循环自动管理 | Dreaming 引擎 + 用户手动编辑 |
| **更新方式** | 每轮对话追加 messages | `update_user_memory` 工具（需用户确认） |

## [m3-hot-cold] 冷热分层：不是工程执念，是 token 预算管理

核心概念：**热会话和冷会话用不同的策略加载历史**。

```
                      ┌──────────────────┐
新消息 ───→ 查上次活跃时间  │ > 2小时？          │
                      └───────┬──────────┘
                              │
                    ┌─────────┴─────────┐
                    │                   │
                    ▼                   ▼
              热会话（Hot）        冷会话（Cold）
         ┌──────────────────┐ ┌──────────────────┐
         │ 加载全部历史      │ │ 1. 生成/复用摘要  │
         │ 消息完整保留      │ │ 2. 只加载最后 8 轮 │
         └──────────────────┘ │ 3. 摘要注入 prompt │
                              └──────────────────┘
```

关键阈值：

| 参数 | 默认值 | 作用 |
|------|--------|------|
| `chat_cold_resume_idle_seconds` | 7200（2小时） | 判定冷/热的空闲时间 |
| `chat_cold_resume_max_turns` | 8 | 冷恢复保留的最大轮数 |
| `chat_hot_summary_turn_threshold` | 50 | 热会话中触发背景摘要的轮数 |
| `chat_force_summary_token_threshold` | 240000 | 强制触发摘要的 token 阈值 |

::: checkpoint
一个用户早上 9 点聊了 60 轮，然后 12 点回来继续聊。他的会话是热还是冷？会发生什么？

--- answer
12 点距离上次活跃 3 小时（> 2h），是**冷会话**。
发生的过程：1) 检查是否有摘要，如果没有则生成摘要；2) 摘要注入系统提示词；3) 只加载最后 8 轮消息。前 52 轮的内容通过摘要"压缩"保留。
---
:::

## [m3-summary] 摘要是怎么做的

当触发摘要时，Agent 用独立的 LLM 调用生成结构化摘要。摘要指令期望的输出格式：

```markdown
当前主题: ...
用户目标: ...
已确认事实:
- ...
关键工具结论:
- ...
未解决事项:
- ...
连续性线索: ...
```

摘要存在 `session_state` 表，下次冷恢复时直接复用，不需要重新生成。

## [m3-dreaming] Dreaming：让 Agent 自我进化

Dreaming 是深小理最特别的设计之一——Agent 不只是被动的，它会在后台"反思"。

### 数据来源：Observations

在对话过程中，Agent 可以通过 `record_observation` 工具记录**自我观察**。每条 observation 分四类：

- `insight`：发现了什么规律
- `feedback`：用户给了什么反馈
- `fact`：确认了什么事实
- `emotion`：感知到了什么情绪

### 双阶段处理

**Phase 1 — 日记生成**（手动触发或定时器）：

1. 取 20 条未处理的 observations
2. 去重（`difflib.SequenceMatcher`，相似度 > 0.80）
3. 按类别分组
4. 加载 SOUL.md / MEMORY.md / SUAT.md 作为锚定上下文
5. LLM（temperature=0.8，鼓励创造性反思）根据 prompt 模板生成日记
6. 写入 `workspace/dairy/YYYY-MM-DD.md`（如已存在则追加"续篇"）

**Phase 2 — 记忆更新**：

1. 读最近 5 篇日记
2. LLM（temperature=0.7）根据模板生成新的 MEMORY.md
3. **备份**当前 MEMORY.md → `.memory_backups/{timestamp}_MEMORY.md`
4. 覆盖写入 `workspace/MEMORY.md`
5. 更新 `changelog.jsonl`（每次变更的审计日志）

::: exercise
追踪一个完整的 Dreaming 双阶段流程。假设深小理今天有 15 条未处理的 observations（8 insight + 4 feedback + 3 emotion），描述 Phase 1 和 Phase 2 各做了什么。每个阶段的关键输入和输出是什么？

--- perspective
依据：基于深小理 Dreaming 引擎的实际实现和 `dreaming/` 目录下的 prompt 模板。

Dreaming 的精妙不在于技术复杂度，在于它模拟了一个"反思"的心理模型：日常积累（observations）→ 周期整理（diary）→ 提炼更新（memory）。这和人类学习的过程高度相似。

但要注意边界：Dreaming 是一个离线过程，不是实时的。它的输出（MEMORY.md）会进入下一次对话的 system prompt，间接影响 Agent 行为。所以 Dreaming 的质量直接决定了 Agent 在长期对话中"长进"多少。
---
:::

## [m3-summary-section] 模块小结

- 两套记忆：session 记忆跟随对话生命周期，持久记忆跨会话存在。
- 冷热分层的本质是 token 预算管理——不是存储问题，是 LLM 上下文窗口约束。
- 摘要用结构化 Markdown 格式，确保关键信息密度。
- Dreaming 引擎是 Agent 的"反思系统"——双阶段 + 备份 + 审计日志，把自我进化做成可控过程。
