## [m8-fit] When ACP fits

ACP fits when the client needs to drive a long-lived or local agent, and that agent may inspect files, request tools, or maintain session context.

Good examples include editor integrations, local development assistants, and controlled desktop tools.

## [m8-overkill] When ACP is overkill

ACP is probably overkill when the application only sends one prompt and receives one answer.

If the task is a stateless web form, an ordinary HTTP API is simpler and easier to operate.

## [m8-framework] Decision framework

Ask four questions:

1. Does the agent need local files or tools?
2. Is the conversation multi-step?
3. Does a client need to mediate tool calls?
4. Will multiple clients or agents benefit from a shared protocol?

If most answers are yes, ACP is worth serious consideration.

## [m8-scenario] Public example: CampusGuide review

Suppose a team is building CampusGuide, a fictional campus Q&A assistant. They want an agent to inspect local docs, run validation scripts, and suggest edits.

That is a plausible ACP scenario: local files matter, tool calls matter, and the client should control what the agent can access.

If the same team only wants to paste one paragraph into a web page and ask for a rewrite, ACP is not necessary.

## [m8-conclusion] Conclusion

ACP is a standardization tool. It is valuable when a standard client-agent boundary reduces real integration pain.

Choose it when it simplifies the system. Skip it when it adds protocol ceremony without improving the learning or development workflow.

## [m8-summary] Module recap

- ACP is strongest for local, multi-step, tool-using agent sessions.
- Simple one-shot chats usually do not need ACP.
- Public teaching examples should be fictitious and safe to publish.
