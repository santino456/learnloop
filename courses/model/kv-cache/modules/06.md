## [m6-when] 什么时候需要优化 KV Cache

不是每个场景都值得投入 KV Cache 的优化。以下是值得投入的信号：

| 信号 | 说明 |
|------|------|
| 多并发服务 | batch size > 1 时，KV Cache 常成为显存瓶颈 |
| 长上下文 | seq_len > 8K 时，KV Cache 增长显著 |
| 显存不足 | 模型能加载，但一跑就 OOM |
| 吞吐量不达标 | GPU 利用率低，但显存满了 |

如果你的场景是单请求、短序列、本地跑 demo，那么标准 Hugging Face 的 `use_cache=True` 已经足够。

> 依据：工程实践观察。单请求短序列下，KV Cache 优化（如 PagedAttention）的管理 overhead 往往超过收益。

## [m6-decision] 优化路径选择

当确定需要优化时，按以下顺序考虑：

1. **GQA/MQA**：如果还在选型阶段，优先选择原生支持 GQA 的模型（如 Llama-2-70B、Mistral）。这是零成本的显存节省。
2. **量化 KV Cache**：把 FP16/BF16 的 KV Cache 量化到 INT8 甚至更低，显存直接减半。代价是轻微的质量下降。依据：多项研究表明 INT8 KV Cache 量化对生成质量影响很小（unverified：具体论文名和版本建议查阅最新文献）。
3. **PagedAttention**：如果运行推理服务，使用 vLLM 或类似框架。这是系统级的优化，不需要改模型。
4. **自定义驱逐策略**：对于超长上下文（>100K），考虑滑动窗口、H2O（Heavy Hitter Oracle）等 token 驱逐策略。依据：这些策略在特定任务上有效，但通用性不如前三种方案稳定。

## [m6-smells] 常见的坏味道

- **过度优化**：在 batch=1、seq_len=512 的场景下折腾 PagedAttention，收益微乎其微。
- **忽视 prefill**：只优化 decode 阶段的 KV Cache，但 prefill 阶段才是用户首次等待的延迟来源。
- **硬编码最大长度**：预分配固定长度的 KV Cache，既不灵活又浪费显存。
- **忽略 dtype**：没意识到 FP32 和 FP16 的 KV Cache 显存差一倍。

> 依据：作者经验与社区常见问题总结。这些坏味道在实际项目中反复出现。

## [m6-ai] 什么时候用 AI 辅助决策

AI（包括 LLM）在 KV Cache 优化中可以帮助：

- **快速估算**：给定模型参数和硬件配置，估算 KV Cache 显存和 batch size 上限。
- **方案对比**：整理不同优化方案的 tradeoff 表格。
- **代码生成**：生成基础的 KV Cache 管理代码或集成 vLLM 的脚本。

但 AI 的输出需要验证：

- **不要轻信性能数字**：AI 可能给出过时的或特定场景下的数字，当作通用结论。
- **确认版本兼容性**：vLLM、Transformers 等库的版本差异很大，AI 的知识可能滞后。
- **实际 benchmark**：任何优化方案都应该在你的实际 workload 上跑 benchmark，而不是凭理论判断。

> 依据：needs-human-review。AI 辅助决策的价值在于加速信息整理，但最终决策应基于实际测量。

## [m6-summary] 模块小结

- KV Cache 优化值得投入的信号：多并发、长上下文、显存瓶颈、吞吐量不达标。
- 优化路径：先选型 GQA 模型 → 量化 KV Cache → PagedAttention → 自定义驱逐策略。
- 常见坏味道：过度优化、忽视 prefill、硬编码最大长度、忽略 dtype。
- AI 可以辅助估算和对比，但关键数字需要实际 benchmark 验证。
