## [m7-why] Why a browser needs a bridge

A browser page cannot safely spawn a local agent process. It also cannot directly manage stdin and stdout pipes.

The usual shape is a local bridge: browser talks to bridge over HTTP or SSE; bridge talks to the ACP agent over stdio.

## [m7-architecture] Bridge architecture

```text
Browser UI
  | HTTP or SSE
Local Bridge
  | JSON lines over stdio
ACP Agent
```

The bridge owns process lifetime, request ids, tool policy, and stream conversion.

## [m7-http] Minimal HTTP responsibilities

A small bridge needs only a few responsibilities at first:

- Accept a user message from the browser.
- Convert it into an ACP request.
- Read agent responses.
- Stream or return the answer.
- Store logs and errors in a debuggable place.

## [m7-sse] HTTP, SSE, or WebSocket

Use plain HTTP for request-response interactions. Use SSE when the main need is streaming text from the agent to the browser. Use WebSocket when both sides need frequent bidirectional messages.

Start with the simplest transport that preserves the learning loop.

## [m7-summary] Module recap

- Browsers need a bridge for local ACP agents.
- The bridge is responsible for process and protocol hygiene.
- SSE is often enough for streaming answers.
- Do not build a full platform before the local loop proves useful.
