## [m4-schema] 数据库 Schema 总览

深小理用 PostgreSQL 14，通过 `psycopg` (v3) 直接操作，**没有 ORM**。所有表在应用启动时通过 `CREATE TABLE IF NOT EXISTS` 自动创建。

连接池：`psycopg_pool.ConnectionPool`，最小 2、最大 25 连接，`dict_row` row factory。

### 10 张表

| 表 | 用途 | 核心字段 |
|----|------|---------|
| `users` | 用户账户（注册 + 访客） | id, username, password_hash, is_guest, is_admin, daily_quota, user_md |
| `conversations` | 所有聊天消息 | id, user_id, session_id, role, content, tool_calls, reasoning_content, images |
| `session_state` | 每会话元数据 | user_id, session_id (PK), summary_text, last_message_at, forked_from |
| `pending_asks` | 暂停的 ReAct 状态 | session_id (PK), messages (JSON), ask_call (JSON) |
| `agent_observations` | Agent 自我观察 | id, content, category, source_session_id, processed |
| `evolution_logs` | 每日增长指标 | date (unique), total_dialogues, new_users |
| `saved_poems` | 用户收藏卡片 | id, user_id, title, content, html, css |
| `posts` | 社区帖（需求/供给） | id, user_id, type, title, content, tags, status |
| `post_tags` | 帖子标签索引 | post_id, tag (composite PK) |
| `feedback` | 用户反馈（like/dislike） | _文件存储，非数据库表_ |

### 索引策略

```sql
-- conversations: 核心查询路径全覆盖
(user_id), (session_id),
(user_id, session_id, timestamp DESC),  -- 会话消息按时间排序
(role, timestamp),                        -- 按角色过滤
(session_id, role, timestamp)            -- 会话+角色组合查询

-- users
(first_seen), (last_seen)

-- agent_observations
(processed, created_at DESC)

-- session_state
(user_id, last_message_at DESC)

-- posts
(user_id, status), (type, status), (created_at DESC)

-- post_tags
(tag), (post_id)
```

> **思考题**：为什么 `conversations` 表需要 (user_id, session_id, timestamp DESC) 这个复合索引？
>
> 因为获取会话消息的查询模式是：`WHERE user_id = ? AND session_id = ? ORDER BY timestamp DESC`。复合索引覆盖了过滤列 + 排序列，避免了 filesort。这是数据库中最常见的查询模式。

## [m4-quota] Quota 原子操作

访客有每日 50 条消息限制（可配置）。消耗配额的 SQL 用原子操作防止竞态：

```sql
UPDATE users
SET total_messages = COALESCE(total_messages, 0) + 1
WHERE id = %s
  AND COALESCE(total_messages, 0) < daily_quota
RETURNING total_messages
```

关键设计：
- `WHERE ... AND total_messages < daily_quota` —— 在 DB 层做条件检查，不用先查再改
- `RETURNING` —— 原子地返回更新后的值，不需要第二条 SQL
- 当天日期变化时自动重置：`quota_date != today` → 重置 total_messages

## [m4-config] 配置系统设计

`backend/app/config.py` 用 `pydantic-settings` 管理配置。几个重要的设计决策：

### 路径解析

```python
_PROJECT_ROOT = Path(__file__).resolve().parent.parent.parent
```

从 config 文件位置推导项目根目录（3 级：config.py → config/ → app/ → backend/ → root），确保绝对路径不受 CWD 影响。

### 向后兼容

旧配置名 `DEEPSEEK_API_KEY` / `MIMO_API_KEY` 通过 `apply_legacy_model_fallbacks()` 自动映射到新的 `primary_llm_*` / `vision_llm_*` 字段。只在目标字段为空时才覆盖。

### 安全守卫

- 启动时拒绝默认 `SECRET_KEY`——直接报错退出
- 生产环境拒绝 `CORS_ALLOW_ORIGINS="*"`——强制指定域名

### 所有可调参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `SECRET_KEY` | (必填) | JWT 签名密钥 |
| `DATABASE_URL` | postgresql://localhost/shenxiaoli | PG 连接串 |
| `CORS_ALLOW_ORIGINS` | http://localhost:3000 | CORS 白名单 |
| `chat_hot_summary_turn_threshold` | 50 | 热会话摘要触发轮数 |
| `chat_cold_resume_idle_seconds` | 7200 | 冷会话判定时间 |
| `chat_cold_resume_max_turns` | 8 | 冷恢复保留轮数 |
| `chat_force_summary_token_threshold` | 240000 | 强制摘要 token 阈值 |
| `chat_semaphore` | 4 | 并发聊天上限 |
| `tts_semaphore` | 2 | 并发 TTS 上限 |
| `guest_daily_quota` | 50 | 访客每日消息数 |

## [m4-security] 安全设计

### Path Traversal 防护

`FileManager` 用 `Path.relative_to()` 确保所有路径操作不逃逸 workspace：

```python
resolved = full_path.resolve()
resolved.relative_to(self.workspace_root)  # 如果逃逸则抛 ValueError
```

### Admin 继承隔离

`AdminShenXiaoLiAgent` 继承 `ShenXiaoLiAgent`，通过**替换** tools 和 skills registry 来获得额外权限（`write_file`、`edit_file`），然后重新编译图。不需要修改核心 Agent 代码。

```python
class AdminShenXiaoLiAgent(ShenXiaoLiAgent):
    def __init__(self):
        super().__init__()  # 构建基础图
        self.tools = AdminToolRegistry()   # 替换为 admin 工具
        self.skills = AdminSkillRegistry()  # 替换为 admin skill
        self.graph = self._build_graph()    # 重新编译
```

### Rate Limiting

基于滑动窗口的内存计数器。5 个 limiter 实例：

| Limiter | Key | 限制 | 作用端点 |
|---------|-----|------|---------|
| login_limiter | `login:{ip}` | 5/min | /auth/login |
| register_limiter | `register:{ip}` | 5/min | /auth/register |
| guest_limiter | `guest:{ip}` | 5/min | /auth/guest |
| chat_limiter | `chat:{user_id}` 或 `chat:{ip}` | 30/min | /chat/* |
| dashboard_limiter | `dashboard:{ip}` | 60/min | /admin/dashboard/* |

**已知弱点**：内存计数器在多 worker 环境下各自独立（不共享）。代码在 import 时检测多 worker 配置并打印警告。生产部署目前是 2 worker，这个问题可控。如果扩展到多 worker，需要迁移到 Redis。

### Privacy Guard

在 Agent 循环中，`privacy_check` 节点在工具执行前拦截敏感调用。用户必须在浏览器确认后才执行。这个设计确保**AI 可以"提议"，但最终决定权在用户**。

## [m4-orm] 为什么不用 ORM

深小理直接用 `psycopg` 写 SQL，没有用 SQLAlchemy 或任何 ORM。

原因：
- 查询模式简单：主要是 CRUD + 少量 JOIN/聚合
- SQL 直接、可调试：`pg_stat_statements` 可以追踪实际查询
- 没有对象映射需求：messages 以 JSONB 存储，不需要映射到 Python 类
- 启动更快：没有 ORM schema 反射

代价：
- 没有 migration 框架——靠 `CREATE TABLE IF NOT EXISTS` + 手动 ALTER
- 切换数据库（如 PostgreSQL → MySQL）需要改写所有 SQL
- 但深小理不需要切换数据库，所以这个代价可以接受

## [m4-summary] 模块小结

- 10 张表 + 密集索引 + 原子 quota 操作 = 简单可靠的持久化层
- 配置系统：pydantic-settings + 向后兼容 + 安全守卫
- 安全层层递进：路径防护 → admin 继承隔离 → rate limit → privacy guard
- ORM 不是必需品：简单查询模式 + 直接 SQL = 更快的启动和更好的可调试性
