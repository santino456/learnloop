## [m2-roles] 客户端与 Agent 的角色

ACP 的通信发生在两个核心角色之间：客户端（Client）与 Agent。

| 角色 | 主要职责 | 典型实例 | 不做什么 |
|------|---------|---------|---------|
| 客户端（Client） | 持有用户界面；组装上下文；发送请求；执行 Agent 申请的工具调用；管理会话生命周期 | 编辑器插件、CLI、本地 Bridge | 不替 Agent 做推理决策 |
| Agent | 接收提示；决定策略；申请工具调用；返回结果；维护内部状态 | 代码补全 Agent、审查 Agent、问答 Agent | 不直接操作用户文件系统或 UI |

责任矩阵：

| 任务 | 客户端 | Agent |
|------|--------|-------|
| 启动 / 终止进程 | ✅ | ❌ |
| 解析用户意图 | 组装输入 | 推理输出 |
| 决定调用哪个工具 | ❌ | ✅ |
| 执行工具调用 | ✅ | ❌ |
| 保证 stdout 干净 | 读取并解析 | 不写入日志 |
| 错误归因 | 捕获并展示 | 返回错误对象 |

> **口诀**：客户端离用户最近，Agent 离推理最近。

> **注意**：不要把 Agent 直接暴露给用户文件系统。客户端作为受信边界，负责执行权限策略。

## [m2-jsonrpc] JSON-RPC 形态

ACP 的消息通常以 JSON-RPC 2.0 帧形式传输。一条最小请求包含四个字段：

| 字段 | 类型 | 必填 | 含义 |
|------|------|------|------|
| `jsonrpc` | string | 是 | 固定为 `"2.0"` |
| `id` | string / number / null | 是 | 请求与响应的对应键 |
| `method` | string | 是 | 调用的方法名 |
| `params` | object / array | 否 | 方法参数 |

示例请求：

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "initialize",
  "params": {
    "capabilities": {},
    "clientInfo": { "name": "minimal-client", "version": "0.1.0" }
  }
}
```

示例响应：

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "protocolVersion": "2025-03-26",
    "capabilities": {}
  }
}
```

错误响应：

```json
{
  "jsonrpc": "2.0",
  "id": 1,
  "error": {
    "code": -32601,
    "message": "Method not found"
  }
}
```

`id` 生成规则：

1. 在同一会话内唯一。
2. 请求与响应的 `id` 必须完全一致。
3. 推荐使用自增整数或 UUID 字符串。
4. `id` 为 `null` 时表示通知，不需要响应。

> **注意**：教学示例字段名可能变化，运行前务必对照真实 Agent 文档。

## [m2-stdio] 本地传输

本地模式下，ACP 使用**行分隔的 JSON** 通过标准流交换消息。

| 流 | 方向 | 内容 | 注意事项 |
|----|------|------|---------|
| stdin | 客户端 → Agent | JSON-RPC 请求 | 每条以 `\n` 结尾 |
| stdout | Agent → 客户端 | JSON-RPC 响应 | 必须只含可解析 JSON |
| stderr | Agent → 客户端 | 日志、诊断、错误 | 客户端不应把它当协议响应 |

Python 最小示例：

```python
import json
import subprocess

proc = subprocess.Popen(
    ["example-agent", "acp"],
    stdin=subprocess.PIPE,
    stdout=subprocess.PIPE,
    stderr=subprocess.PIPE,
    text=True,
    bufsize=1,
)


def send(msg):
    proc.stdin.write(json.dumps(msg) + "\n")
    proc.stdin.flush()


def receive():
    line = proc.stdout.readline()
    if not line:
        raise RuntimeError("agent closed stdout")
    return json.loads(line)
```

缓冲细节：

- `text=True` 让 Python 以文本模式读写，方便处理 JSON 字符串。
- `bufsize=1` 开启行缓冲，配合显式 `flush()` 使用。
- 每次 `send` 后必须 `flush()`，否则消息可能滞留在缓冲区。
- 读取端使用 `readline()` 阻塞等待下一行；若需并发处理，应在独立线程 / 协程中读取。

常见错误：

| 错误 | 后果 | 修复 |
|------|------|------|
| 日志写入 stdout | JSON 解析失败 | 所有日志改走 stderr |
| 忘记 flush | Agent 永远等不到输入 | 每次 write 后 flush |
| 使用二进制模式不拆行 | 帧边界丢失 | text=True 并自行加 `\n` |
| stderr 与 stdout 合并 | 日志混入响应 | 分别捕获两条流 |

> **注意**：stdout 一旦混进日志，客户端解析 JSON 行就会失败。日志必须走 stderr。

## [m2-lifecycle] 会话生命周期

一个完整会话可以拆成以下步骤：

1. **启动 Agent 子进程**
   - 客户端根据配置或用户命令启动进程。
   - 确认进程已准备好读取 stdin。
2. **初始化握手**
   - 发送 `initialize` 请求。
   - 等待 `initialize` 响应，确认协议版本与能力。
   - 若握手失败，立即终止会话并报告错误。
3. **发送任务请求**
   - 将用户输入组装成 prompt 或任务消息。
   - 附加必要的资源引用与上下文。
4. **处理响应**
   - 若响应包含最终答案，直接展示。
   - 若响应包含工具调用申请，客户端执行并返回 `tool_result`。
   - 重复直到 Agent 返回最终答案或错误。
5. **错误处理**
   - JSON-RPC 错误：按 `code` 分类（解析错误、方法不存在、应用错误）。
   - 传输错误：进程崩溃、stdout 关闭、超时。
   - 能力不匹配：握手阶段即应失败。
6. **关闭进程**
   - 发送退出信号或关闭 stdin。
   - 等待进程退出，收集 stderr 日志用于排错。

```text
Client          Agent
  | initialize  |
  |-----------> |
  |  result     |
  | <-----------|
  | prompt/task |
  |-----------> |
  | tool_call   |
  | <-----------|
  | tool_result |
  |-----------> |
  | response    |
  | <-----------|
  | shutdown    |
  |-----------> |
```

> **注意**：错误处理越早越好。不要在握手失败后继续发送业务请求。

## [m2-summary] 模块小结

- ACP 把面向用户的客户端职责和 Agent 的推理职责分开。
- 本地 ACP 依赖干净的 stdout 和显式刷新缓冲区。
- JSON-RPC `id` 是请求 / 响应匹配的关键。
- 示例只是协议草图，必须针对具体 Agent 版本验证后再运行。

快速参考卡：

- **客户端**：组装上下文 + 执行工具 + 管理会话。
- **Agent**：接收提示 + 推理 + 申请工具。
- **stdin / stdout**：一次一行 JSON；stderr 只放日志。
- **生命周期**：启动 → 握手 → 任务 → 响应 / 工具循环 → 关闭。
