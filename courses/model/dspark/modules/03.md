## [m3-problem] 盲目验证的浪费

::: concept
title: 并行 draft 带来长候选，但 batch 容量有限
why: 生产环境里不是所有候选 token 都值得用大模型验证。

并行 draft 能一次性提出 7 个甚至更多候选 token，但目标模型一次 forward 能验证的 token 数量受 batch capacity 限制。如果不管三七二十一全部送进去验证：

- 低置信度的尾部 token 大概率被拒绝，白白占用 batch 容量。
- 高并发时，浪费的容量会拖慢所有请求的响应。

论文把这种现象称为 **indiscriminate verification**，是并行 draft 在高并发 serving 中的主要系统瓶颈。
:::

## [m3-confidence] 置信度头与调度器

::: concept
title: Confidence head 估计每个前缀的生存概率
why: 需要提前判断哪些候选 token 更可能被接受。

DSpark 在 draft 模型上加了一个轻量的 **confidence head**。对每个候选位置 $j$，它输出一个标量置信度 $c_{r,j}$，表示该 token 被目标模型接受的概率。

基于这些置信度，可以计算前缀生存概率：

$$
a_{r,j} = \prod_{i \le j} c_{r,i}
$$

即前 $j$ 个 token 全部被接受的概率。这个概率越高，继续往后验证的收益越大。
:::

::: concept
title: Hardware-Aware Prefix Scheduler 动态分配验证长度
why: 验证长度应该随负载变化，而不是固定值。

DSpark 的调度器会：

1. 把所有请求的所有候选前缀按生存概率全局排序。
2. 根据当前 GPU 吞吐曲线（预先用不同 batch size profile 得到）计算最优 batch size $B$。
3. 逐步从高到低 admitting token，直到达到容量 $K$。

负载轻时，可以接受更多长前缀，充分利用空闲算力；负载重时，只验证高置信度前缀，保护 batch 容量。论文把这个机制叫做 **Confidence-Scheduled Verification**。
:::

::: evidence
claim: Confidence-Scheduled Verification 在高并发下能显著提升系统吞吐，避免低置信度 token 浪费 batch 容量。
source: [DSpark 论文 PDF](https://github.com/deepseek-ai/DeepSpec/blob/main/DSpark_paper.pdf)
status: verified
basis: 论文 Section 3.2 与 Figure 7 显示，在严格延迟 SLA 下 DSpark 相对 MTP-1 的吞吐提升可达 406–661%，调度器是关键系统优化之一。
:::

## [m3-training-objective] 训练目标

::: concept
title: 总变差损失直接优化接受率
why: 让 draft 产出的 token 更容易被目标模型接受。

DSpark 的训练目标由三项加权组成：

$$
\mathcal{L} = \alpha_{\text{ce}} \mathcal{L}_{\text{ce}} + \alpha_{\text{tv}} \mathcal{L}_{\text{tv}} + \alpha_{\text{conf}} \mathcal{L}_{\text{conf}}
$$

默认权重为 $\alpha_{\text{ce}}=0.1$、$\alpha_{\text{tv}}=0.9$、$\alpha_{\text{conf}}=1.0$。

- $\mathcal{L}_{\text{ce}}$：标准交叉熵，保证 draft 分布合理。
- $\mathcal{L}_{\text{tv}}$：总变差损失（total-variation loss），让 draft 分布与目标模型分布更接近，从而提升接受率。
- $\mathcal{L}_{\text{conf}}$：置信度头损失，让 confidence head 的估计更准确。
:::

::: exercise
假设某请求有 5 个候选 token，置信度分别为 0.95、0.90、0.85、0.60、0.30。请计算前 3 个 token 全部 survive 的概率，并判断是否应该验证第 5 个 token。

--- answer
前 3 个 token 全部 survive 的概率 = 0.95 × 0.90 × 0.85 ≈ 0.727。
第 5 个 token 的前缀 survive 概率 = 0.95 × 0.90 × 0.85 × 0.60 × 0.30 ≈ 0.131。在 batch 容量紧张时不值得验证，应优先把容量留给更高置信度的前缀。
---
:::

## [m3-summary] 本章小结

- 并行 draft 的长候选如果全部验证，会浪费宝贵的 batch 容量。
- Confidence head 给每个 token 估计接受概率，前缀生存概率指导调度。
- Hardware-Aware Prefix Scheduler 根据负载动态选择验证长度，实现高并发下的吞吐最大化。
- 训练目标同时优化 draft 质量、分布一致性和置信度估计。
