## [m2-why] ReAct 是什么，为什么用它

ReAct = Reasoning + Acting。AI 不是一口气回答，而是：**先想 → 做（调工具）→ 根据结果再想 → 再回答**。

深小理的核心是一个 LangGraph `StateGraph`，编译成有向图后执行。选择 LangGraph 而不是自己写 while 循环，有三个原因：

1. **状态管理**：每条消息、每次工具调用结果都在 `AgentState` 里精确追踪
2. **中断/恢复**：privacy confirm 需要暂停等待用户确认，LangGraph 原生支持
3. **可可视化**：图结构让调试时有清晰的心智模型

## [m2-graph] 图的完整结构

```
START → prepare → model → (router) ─┬→ END (直接回复)
                                    └→ privacy_check → (router) ─┬→ privacy_confirm → END
                                                                  └→ tools → (router) → model → ...
                                                                                       └→ ask_user → END
                                                                                       └→ error → END
```

7 个节点，4 个条件路由：

| 节点 | 职责 | 触发条件 |
|------|------|---------|
| `prepare` | 组装系统提示词（soul + SUAT + memory + skills + tool guidelines） | 每次新消息 |
| `model` | 调用 LLM，返回文本或 tool_calls | prepare 之后 |
| `privacy_check` | 检查 tool_calls 是否包含敏感工具 | model 返回了 tool_calls |
| `privacy_confirm` | 格式化确认问题发给用户 | 检测到隐私工具调用 |
| `tools` | 并行执行所有 tool calls | 不在隐私确认中 |
| `ask_user` | 暂停等待用户输入 | tool 返回了 ask_user 类型 |
| `error` | 友好错误消息 | tool 执行异常 |

## [m2-prepare] prepare：组装系统提示词

每次新消息进来，`prepare` 节点动态构建系统提示词，包含：

```
SOUL.md (人格) → SUAT.md (校园事实) → MEMORY.md (经验) →
interaction_policy.md (交互原则) → tool_guidelines.md (工具规范) →
skills manifest (可用技能清单) → session summary (长对话摘要) →
user memory (用户记忆文档)
```

关键设计：**构建时过滤掉旧 system 消息**——因为 ReAct 循环会多次经过 prepare，不能让 system prompts 重复累积。

::: checkpoint
为什么 ReAct 循环中 system 消息会重复累积？不处理会有什么后果？

--- answer
因为 prepare 节点每次进入都会重新构建 system prompt，如果旧的 system 消息不清除，消息列表会每隔一次循环就多一条 system 消息。后果：token 开销指数级增长，LLM 可能被多条 system 指令混淆。
---
:::

## [m2-model] model：调用 LLM 并处理返回

`model` 节点做三件事：

1. **调用 LLM**：带 3 次重试（1-2 秒退避），temperature=0.7，max_tokens=16000，超时 90s
2. **保留 reasoning_content**：DeepSeek/MiMo 的思考内容（thinking 模式），对多轮工具调用很关键
3. **工具调用上限**：`loop_count >= 10` 时强制截断，不再发出 tool calls

## [m2-tools] tools：并行执行

`tools` 节点使用 `asyncio.gather()` 并行执行所有工具调用。每个 tool 有独立的 `_run_one()` 协程，互不阻塞。

工具结果会构建为 `ToolMessage` 加入消息列表。每个工具类型有自定义的紧凑预览：

| 工具 | 预览格式 | 为什么 |
|------|---------|--------|
| `read_file` | "读取了 N 字符" | 全文太长，前端不需要展示 |
| `search` | "找到 N 条结果" | 同上 |
| `list_files` | "N 个文件和目录" | 同上 |
| `execute` | "脚本执行完成" | 同上 |
| `ask_user` | "{{question}}" | 问题本身要展示 |
| `record_observation` | "已记录观察" | 隐私工具，不暴露内容 |
| `update_user_memory` | "已更新用户记忆" | 同上 |

::: checkpoint
重看 tool 预览的设计。为什么要做紧凑预览而不是直接把 tool 返回值扔给前端？换个说法：tool result 里面的信息，前端用户真的需要看到吗？

--- answer
Tool result 是给 Agent 的"草稿纸"，不是给用户看的报告。把所有 tool 返回值扔给前端会：1) 增加 SSE 带宽消耗；2) 让聊天界面变得混乱；3) 暴露内部文件路径/内容给用户。紧凑预览在前端展示"Agent 在做事"的信号就够了。这个设计背后是一个原则：Agent 的内部工作不要全部暴露给用户，只展示那些用户关心的信号。
---
:::

## [m2-privacy] privacy：中断与确认

两个工具被标记为隐私敏感：`record_observation` 和 `update_user_memory`。

当模型提出要调用这些工具时，`privacy_check` 节点拦截，然后 `privacy_confirm` 发确认请求给用户。**图在此处中止**。

用户点击"允许"或"拒绝"后，前端调 `/chat/privacy/stream` 恢复：
- 允许 → 执行工具 → 继续 ReAct 循环
- 拒绝 → 告知模型被拒绝 → 继续下一个思考

::: checkpoint
如果不用中断模式，而是让 Agent 直接写入 memory 再事后告知用户，会有什么问题？

--- answer
核心问题是 AI 的判断不可靠。让 Agent 自主修改用户记忆意味着：1) 错误推断会被写入"持久记忆"；2) 用户没有修正机会；3) 长期累积后记忆会越来越偏离用户真实意图。中断确认是信任边界——AI 可以"提议"，用户做最终决定。
---
:::

## [m2-session-lock] 会话锁：并发控制

每个 session 一个 `asyncio.Lock`，防止同个 session 的并发消息导致状态混乱。

LLM 调用期间 node 在 IO 等待（async），但同一 session 只有一条消息在处理。不同 session 之间完全并行。

锁用 LRU-bounded `OrderedDict`（max 1024 条）管理，防止长时间不活跃的 session 锁占用内存。

## [m2-summary] 模块小结

- 深小理用 LangGraph StateGraph 建模 ReAct 循环——不是简单的 while 轮询。
- 7 个节点各司其职：prepare（提示词）→ model（LLM）→ privacy_check（检测）→ tools（执行）→ model（再想）→ 回答。
- 关键安全设计：privacy 中断、loop_count 硬上限（10）、紧凑 tool 预览。
- 会话锁（per-session asyncio.Lock）防止并发混乱。
