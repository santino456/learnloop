## [m2-roles] 客户端与 Agent 的角色

在 ACP 里，**客户端**负责用户能直接触摸到的界面：发送请求、提供上下文、并代表 Agent 执行工具调用。

**Agent**则负责推理和任务执行策略：接收提示、申请工具调用、返回结果。

## [m2-jsonrpc] JSON-RPC 形态

很多 ACP 示例采用类似 JSON-RPC 的消息：每条请求有 `id`、`method` 和可选的 `params`；每条响应通过相同的 `id` 与请求对应。

```json
{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"capabilities":{}}}
```

在教学或实现时，一定要对照你实际运行的 Agent 版本确认字段名称。

## [m2-stdio] 本地传输

本地模式通常是**行分隔的 JSON**，通过标准输入和标准输出传输。

- 客户端向 Agent 的 stdin 写入一条 JSON 并加上换行。
- Agent 向 stdout 写入一条 JSON 并加上换行。
- 日志应该输出到 stderr，保证 stdout 始终可解析。

## [m2-lifecycle] 会话生命周期

一次最小会话通常遵循这样的心智模型：

1. 启动 Agent 进程。
2. 发送 initialize 请求。
3. 发送 prompt 或任务请求。
4. 处理响应消息以及可能的工具调用。
5. 会话结束时关闭进程。

::: checkpoint
为什么 stdout 必须保持“干净”？如果日志混进 stdout 会发生什么？
:::

## [m2-summary] 模块小结

- ACP 把面向用户的客户端职责和 Agent 的推理职责分开。
- 本地 ACP 依赖干净的 stdout 和显式刷新缓冲区。
