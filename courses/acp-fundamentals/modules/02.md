## [m2-roles] 客户端与 Agent 的角色

| 角色 | 职责 |
|------|------|
| 客户端（Client） | 持有用户界面，发送请求，提供上下文，代表 Agent 执行工具调用 |
| Agent | 接收提示，决定策略，申请工具调用，返回结果 |

> 口诀：**客户端离用户最近，Agent 离推理最近**。

## [m2-jsonrpc] JSON-RPC 形态

请求与响应通过同一个 `id` 对应。

```json
{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"capabilities":{}}}
```

| 字段 | 含义 |
|------|------|
| `jsonrpc` | 协议版本标识 |
| `id` | 请求与响应的对应键 |
| `method` | 调用的方法名 |
| `params` | 方法参数 |

> 教学示例字段名可能变化，运行前务必对照真实 Agent 文档。

## [m2-stdio] 本地传输

本地模式使用**行分隔的 JSON**。

- 客户端 → Agent `stdin`：一条 JSON + 换行。
- Agent → 客户端 `stdout`：一条 JSON + 换行。
- 日志必须输出到 `stderr`，保证 `stdout` 始终可解析。

## [m2-lifecycle] 会话生命周期

1. 启动 Agent 子进程。
2. 发送 `initialize` 请求。
3. 发送 prompt 或任务请求。
4. 处理响应与可能的工具调用。
5. 关闭进程。

> 注意：stdout 一旦混进日志，客户端解析 JSON 行就会失败。日志必须走 stderr。

## [m2-summary] 模块小结

- ACP 把面向用户的客户端职责和 Agent 的推理职责分开。
- 本地 ACP 依赖干净的 stdout 和显式刷新缓冲区。
- 示例只是协议草图，必须针对具体 Agent 版本验证后再运行。
