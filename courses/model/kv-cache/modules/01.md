## [m1-confusion] 一个反直觉的问题

你已经知道 Transformer 用自注意力让每个 token "看到" 序列中所有其他 token。但当你用模型生成文本时，有一个细节很容易忽略：

> 生成第 5 个 token 时，前 4 个 token 的 K 和 V 其实**已经被算过一遍了**——在生成第 4 个 token 的时候。

生成第 6 个 token 时，前 5 个 token 的 K 和 V 又要重算。序列越长，这种重复计算越严重。

这就是 KV Cache 要解决的起点：自回归生成中，前面 token 的 K、V 被反复重算，计算量随序列长度平方增长。

::: checkpoint
自回归生成中，生成第 N 个 token 时，前 N-1 个 token 的 K 和 V 矩阵发生了什么？

--- answer
它们被重新计算了，尽管在生成第 N-1 个 token 时就已经算过一次。这种重复计算是 KV Cache 要解决的核心问题。
---
:::

## [m1-solution] KV Cache 的回答

KV Cache 的思路很简单：**把已经算过的 K 和 V 存起来，下一步直接复用。**

具体来说，每生成一个新 token，只需要：

1. 计算这个新 token 自己的 Q、K、V。
2. 把新 token 的 K 和 V 追加到缓存中。
3. 用新 token 的 Q 去和缓存中**所有** K 做注意力计算。

这样，前面 token 的 K、V 只需要算一次，后续步骤直接从缓存读取。

> KV Cache 用空间换时间：把中间结果存下来，避免重复计算。

::: checkpoint
KV Cache 用什么换取了计算效率的提升？

--- answer
用显存空间换取计算时间的减少。缓存存储了已计算 token 的 K 和 V 矩阵，避免每一步都重新计算。
---
:::

## [m1-impact] 效果有多大

不加 KV Cache 时，生成长度为 N 的序列，注意力部分的计算量是 O(N² × d)，其中 d 是头维度。加了 KV Cache 后，每生成一个新 token 只需要 O(N × d) 的计算量，总计算量降到 O(N² × d)，但常数因子大幅减小——因为每一步只算新 token 的 Q 和所有缓存的 K 的点积。

更关键的是，不加 KV Cache 时生成第 N 个 token 需要重新前向传播 N-1 个 token 的全部变换，而加了 KV Cache 后只需要前向传播 1 个 token。这在长序列生成中差距极大。

## [m1-tradeoff] 代价是什么

KV Cache 不是免费的。它带来的主要代价是**显存占用**：

- 缓存随序列长度线性增长。
- 多层 Transformer 每层都要存自己的 K 和 V。
- 多头注意力下，每个头都有独立的 K 和 V。

在多并发、长序列场景下，多个请求的 KV Cache 总量可能达到数十甚至上百 GB——可能超过模型权重本身。

::: checkpoint
KV Cache 的显存占用和哪些因素成正比？

--- answer
和序列长度、Transformer 层数、注意力头数、头维度（以及 batch size）成正比。序列越长、层越多、头越多，KV Cache 占用越大。
---
:::

## [m1-not] KV Cache 不是什么

KV Cache 不是模型权重的缓存。模型权重在推理期间不变，不需要每步更新。KV Cache 缓存的是**激活值**——每一层在处理输入序列时产生的中间结果。

KV Cache 也不是通用的中间结果缓存。它只缓存注意力机制中的 K 和 V，不缓存前馈网络（FFN）的中间结果，因为 FFN 对每个 token 是独立计算的，没有跨 token 的依赖。

## [m1-summary] 模块小结

- 自回归生成导致前面 token 的 K、V 被反复重算。
- KV Cache 把已计算的 K、V 存起来，用空间换时间。
- 代价是显存占用随序列长度线性增长。
- KV Cache 缓存的是激活值，不是模型权重。

::: checkpoint
以下关于 KV Cache 的说法，哪一项是正确的？

A. 缓存模型权重  
B. 缓存 K 和 V 激活值  
C. 缓存 FFN 中间结果  
D. 只在训练时有用

--- answer
B。KV Cache 缓存的是注意力机制中已计算 token 的 K 和 V 矩阵（激活值），用来避免自回归生成中的重复计算。模型权重不需要缓存，FFN 中间结果也没有跨 token 依赖。
---
:::
