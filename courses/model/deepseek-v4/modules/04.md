## [m4-context] 为什么需要 CSA

当上下文达到百万 token 时，标准注意力成为主要瓶颈：

- 计算量随序列长度二次增长。
- KV Cache 随序列长度线性增长，显存压力巨大。

Compressed Sparse Attention（CSA）的解决思路是**先压缩，再稀疏**：

1. 把每 m 个 token 的 KV 压缩成 1 个条目。
2. 再用 DeepSeek Sparse Attention（DSA）选择 top-k 个压缩 KV 条目进行核心注意力。

> 来源：[DeepSeek-V4 论文](raw/DeepSeek_V4.pdf)，第 2.3.1 节。

## [m4-compression] KV 压缩

给定输入隐藏状态 H ∈ R^(n×d)，CSA 先计算两组 KV 条目 C_a、C_b 和对应的压缩权重 Z_a、Z_b：

```
C_a = H · W_a_KV
C_b = H · W_b_KV
Z_a = H · W_a_Z
Z_b = H · W_b_Z
```

然后每 m 个条目压缩成一个条目 C^Comp_i。压缩时同时考虑当前块和前一块的尾部（重叠设计），因此实际上序列长度被压缩到 **1/m**。

::: checkpoint
CSA 的压缩比率大约是多少？

--- answer
每 m 个 token 压缩为 1 个 KV 条目，序列长度约变为原来的 1/m。
---
:::

## [m4-indexer] Lightning Indexer 稀疏选择

压缩后，CSA 用 DSA 策略选择 top-k 个压缩 KV 条目。选择过程叫 Lightning Indexer：

1. 对查询 token t，先通过下投影 W_DQ 得到压缩查询向量 c^Q_t。
2. 再用 W_IUQ 上投影得到 n_I_h 个 indexer query heads。
3. 计算查询与每个压缩块的 index score：
   - 用 ReLU 激活的 indexer query 与压缩 indexer key 做点积。
   - 多个 head 的分数加权求和。
4. 保留 top-k 个压缩 KV 条目进入核心注意力。

::: flow
查询 token t
  -> 下投影得到 c^Q_t
  -> 上投影得到多个 indexer queries
  -> 与压缩 indexer keys 计算 index scores
  -> Top-k 选择压缩 KV 条目
  -> 进入核心 attention
:::

> 来源：[DeepSeek-V4 论文](raw/DeepSeek_V4.pdf)，第 2.3.1 节。

## [m4-mqa] 共享 KV MQA 与分组输出投影

选定压缩 KV 条目后，CSA 以 **Multi-Query Attention（MQA）** 方式执行核心注意力：每个压缩 KV 条目同时作为 key 和 value，所有 query head 共享同一组 KV。

为了减少大 head 维度带来的输出投影开销，CSA 使用**分组输出投影**：

1. 把 n_h 个输出分成 g 组。
2. 每组先投影到较低维度的中间输出。
3. 再把所有组拼接并投影到最终 d 维输出。

::: checkpoint
CSA 核心注意力为什么能用 MQA？

--- answer
因为 KV 已经被压缩成少量共享条目，天然适合让所有 query head 共享同一组 KV，进一步减少计算和显存。
---
:::

## [m4-summary] 模块小结

- CSA = 压缩 + 稀疏索引 + 共享 KV MQA。
- 先把序列长度压缩到 1/m，再用 Lightning Indexer 选择 top-k 个压缩块。
- 分组输出投影用于降低大 head 维度下的输出投影成本。
- CSA 是 DeepSeek-V4 在 1M token 场景下保持高效率的关键之一。
