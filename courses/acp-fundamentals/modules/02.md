## [m2-roles] Client and agent roles

In ACP, the client owns the user-facing surface. It sends requests, provides context, and may execute tools on behalf of the agent.

The agent owns reasoning and task execution strategy. It receives prompts, may ask for tool calls, and returns responses.

## [m2-jsonrpc] JSON-RPC shape

Many ACP examples use JSON-RPC-like messages: each request has an `id`, a `method`, and optional `params`; each response links back to the request id.

```json
{"jsonrpc":"2.0","id":1,"method":"initialize","params":{"capabilities":{}}}
```

When teaching or implementing from an example, verify the exact fields against the agent you are running.

## [m2-stdio] Local transport

The local pattern is line-delimited JSON over standard input and output.

- The client writes one JSON object plus a newline to the agent's stdin.
- The agent writes one JSON object plus a newline to stdout.
- Logs should go to stderr so stdout stays parseable.

## [m2-lifecycle] Conversation lifecycle

A minimal session usually follows this mental model:

1. Start the agent process.
2. Send an initialize request.
3. Send a prompt or task request.
4. Handle response messages and possible tool calls.
5. Close the process when the session is done.

## [m2-summary] Module recap

- ACP separates user-facing client responsibilities from agent responsibilities.
- Local ACP depends on clean stdout and explicit flushing.
- Treat examples as protocol sketches until they are tested against the exact agent version.
