## [m8-default-permissions] 默认权限模型

Pi 没有内置权限系统。默认情况下，它拥有启动用户和进程的全部权限：可以读写文件、执行任意命令、访问网络、读取环境变量中的凭证。

这意味着：

- 不要把 Pi 当成一个默认安全的沙箱；
- 来自不可信来源的 extension、skill、prompt template 都可能执行任意代码；
- 在敏感项目或生产环境中，必须显式做隔离。

> 来源：[packages/coding-agent/docs/containerization.md](https://github.com/earendil-works/pi/blob/371adcf37130629ffb9bbeed9f5548ce08ffa93b/packages/coding-agent/docs/containerization.md) 与 [Pi README — Permissions & Containerization](https://github.com/earendil-works/pi/blob/371adcf37130629ffb9bbeed9f5548ce08ffa93b/README.md#permissions--containerization)。

::: checkpoint
Pi 默认是否提供文件系统/进程/网络隔离？

--- answer
否。Pi 默认以启动用户权限运行，没有内置权限系统。
---
:::

## [m8-containerization] 三种容器化方案

Pi 文档推荐了三种隔离模式：

| 模式 | 隔离对象 | 最佳场景 |
|---|---|---|
| **Gondolin extension** | 内置工具和 `!` 命令 | 本地微 VM 隔离，同时把 auth 留在宿主机 |
| **Plain Docker** | 整个 `pi` 进程 | 简单本地隔离 |
| **OpenShell** | 整个 `pi` 进程 | 策略控制的本地或远程沙箱 |

### Gondolin

[Gondolin](https://github.com/earendil-works/gondolin) 是一个本地 Linux micro-VM。Pi 提供示例 extension，把 `read`、`write`、`edit`、`bash`、`grep`、`find`、`ls` 以及 `!` 命令都路由到 VM 内，同时 API key 仍留在宿主机。

### Plain Docker

把整个 `pi` 进程放进 Docker 容器，API key 通过环境变量传入。适合快速搭建隔离环境。

### OpenShell

NVIDIA OpenShell 提供策略控制的沙箱，可以限制文件系统、进程、网络、凭证和推理访问。适合需要细粒度策略或远程管理的场景。

> 来源：[packages/coding-agent/docs/containerization.md — Choose a pattern](https://github.com/earendil-works/pi/blob/371adcf37130629ffb9bbeed9f5548ce08ffa93b/packages/coding-agent/docs/containerization.md#choose-a-pattern)。

::: checkpoint
哪种方案能让 Pi 主体在宿主机运行，但把危险操作路由到隔离环境？

--- answer
Gondolin extension。
---
:::

## [m8-supply-chain] 供应链加固

Pi 对 npm 依赖的处理非常谨慎：

- 直接外部依赖钉死到精确版本；
- `.npmrc` 设置 `save-exact=true` 和 `min-release-age=2`；
- `package-lock.json` 是唯一依赖真相；
- 预提交钩子默认阻止 lockfile 提交，除非 `PI_ALLOW_LOCKFILE_CHANGE=1`；
- `npm run check` 会验证直接依赖是否钉死、TypeScript 导入是否兼容、shrinkwrap 是否最新；
- 发布包包含 `npm-shrinkwrap.json`，让 npm 用户的传递依赖也被钉死；
- 发布前做本地 smoke test（Node 和 Bun 安装）；
- CI 使用 `npm ci --ignore-scripts`，并定期跑 `npm audit`。

> 来源：[Pi README — Supply-chain hardening](https://github.com/earendil-works/pi/blob/371adcf37130629ffb9bbeed9f5548ce08ffa93b/README.md#supply-chain-hardening)。

::: checkpoint
为什么 `packages/coding-agent/npm-shrinkwrap.json` 对用户很重要？

--- answer
它把 Pi 的传递依赖也钉死，确保 npm 用户安装时不会引入未经审核的新版本。
---
:::

## [m8-perspective] 二次开发决策框架

基于前面的内容，什么时候应该选择哪种二次开发路径？

### 使用 Extension

适合：

- 增加自定义 tool（如调用内部 API、执行项目脚本）；
- 增加项目特定命令或快捷键；
- 做权限门、审计、安全检查；
- 自定义 compaction 或 UI。

风险：

- extension 以你的权限运行，代码来自不可信来源时要小心；
- 项目级 extension 只在信任后加载。

### 使用 Skill

适合：

- 封装领域工作流程或检查清单；
- 共享提示词模板；
- 不需要改运行时的场景。

优势：

- 不直接执行代码，风险低于 extension；
- 可以跨 agent harness 共享（符合 Agent Skills 标准）。

### Fork / 改源码

适合：

- 要改 agent loop 核心行为（ steering/follow-up 队列策略、context transform 等）；
- 要替换或重写 `pi-tui` 的渲染层；
- 要长期维护一个与 Pi 主路线不同的分支。

代价：

- 需要跟踪上游更新；
- 发布和维护成本更高。

### 另起炉灶，只复用底层包

适合：

- 你要做一个完全不同的产品形态（Web IDE、后台服务、自动化流水线）；
- 不需要 Pi 的 TUI 和 CLI。

优势：

- 可以只取 `pi-ai` 和 `pi-agent-core`，自己搭上层；
- 不受 Pi CLI 的假设约束。

::: checkpoint
如果你只是想给团队共享一套“代码审查检查清单”，应该优先用 Extension 还是 Skill？

--- answer
Skill。它不需要改运行时，且符合 Agent Skills 标准，容易跨工具共享。
---
:::

## [m8-security-practices] 安全实践清单

基于以上，二次开发时建议遵循：

1. **审查所有 extension/skill 来源**，不要安装不可信包。
2. **敏感项目使用 Gondolin/Docker/OpenShell 隔离**。
3. **项目级扩展只在信任后加载**；不要给任意项目自动信任。
4. **自定义 tool 参数用 TypeBox 严格校验**，不要相信模型输出的参数。
5. **避免在 extension 中硬编码凭证**，优先用环境变量或 credential store。
6. **发布 Pi Package 时钉死依赖**，并跑本地 smoke test。
7. **定期关注上游安全公告和 AGENTS.md 更新**。

> 来源：本地 m6《Extensions：扩展 Pi 的行为》、m7《Skills 与 Pi Packages》；外部 [Pi README — Supply-chain hardening](https://github.com/earendil-works/pi/blob/371adcf37130629ffb9bbeed9f5548ce08ffa93b/README.md#supply-chain-hardening)、[packages/coding-agent/docs/containerization.md](https://github.com/earendil-works/pi/blob/371adcf37130629ffb9bbeed9f5548ce08ffa93b/packages/coding-agent/docs/containerization.md)。

## [m8-summary] 模块小结

- Pi 默认没有权限隔离，必须以用户权限运行，因此 extension/skill 来源必须可信。
- 三种推荐隔离方案：Gondolin（工具路由到 VM）、Docker（整个进程容器化）、OpenShell（策略控制沙箱）。
- Pi 的供应链策略非常严格：钉死依赖、lockfile 保护、shrinkwrap、ignore-scripts、发布前 smoke test。
- 二次开发路径选择：Extension 改运行时、Skill 封装提示词工作流、Fork 改核心、只复用底层包做新产品。
- 安全实践：审查来源、隔离敏感项目、严格校验 tool 参数、不硬编码凭证、钉死发布依赖。

> 来源：本地 m1–m7；外部 [Pi README](https://github.com/earendil-works/pi/blob/371adcf37130629ffb9bbeed9f5548ce08ffa93b/README.md)、[packages/coding-agent/docs/containerization.md](https://github.com/earendil-works/pi/blob/371adcf37130629ffb9bbeed9f5548ce08ffa93b/packages/coding-agent/docs/containerization.md)。
