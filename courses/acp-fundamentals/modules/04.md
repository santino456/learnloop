## [m4-prompt] Prompt content

A prompt is not just a string. In protocol-oriented systems it often becomes a structured message with a role and one or more content blocks.

```json
{
  "role": "user",
  "content": [
    {"type": "text", "text": "Summarize this project for a new contributor."}
  ]
}
```

## [m4-resources] Resource references

Resources point the agent at context without pasting everything into a single prompt. For a public teaching example, use a fictitious project path:

```json
{
  "type": "resource_link",
  "uri": "file:///workspace/campus-guide/README.md",
  "description": "CampusGuide project README"
}
```

CampusGuide is a fictional campus Q&A assistant. It is useful as an example because it has documents, user questions, and local files without exposing any private project.

## [m4-tool-flow] Tool call flow

An agent may decide it needs a tool result before answering. The client decides whether the tool call is allowed, executes it, and returns the result.

```json
{
  "name": "read_file",
  "arguments": "{\"path\":\"/workspace/campus-guide/README.md\"}"
}
```

The agent should not silently assume tool results. The client should not execute unsafe tool calls without a policy.

## [m4-example] Complete teaching example

A learner asks: "What does CampusGuide do?"

The agent asks to read `/workspace/campus-guide/README.md`. The client returns a short file excerpt. The agent answers: "CampusGuide is a fictional campus Q&A assistant used in this lesson to explain resource and tool-call flow."

## [m4-summary] Module recap

- Prompt messages carry user intent.
- Resources make context addressable.
- Tool calls let the agent ask the client to perform controlled work.
- Public examples should use fictitious paths and projects.
