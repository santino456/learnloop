## [m8-fit] 什么时候适合 ACP

ACP 适合客户端需要驱动一个长期运行或本地的 Agent，而这个 Agent 可能需要检查文件、申请工具、或维护会话上下文。

好的例子包括：编辑器插件、本地开发助手、受控的桌面工具。

## [m8-overkill] 什么时候 ACP 是大材小用

如果你的应用只发一个 prompt、收一个回答，ACP 大概率是过度设计。

对于一个无状态的网页表单来说，普通 HTTP API 更简单、也更容易运维。

::: checkpoint
“单轮问答”和“多轮工具调用会话”之间，哪一个是 ACP 的甜点区？

--- answer
多轮工具调用会话。单轮问答通常不需要 ACP 的协议开销。
---
:::

## [m8-framework] 决策四问

问自己四个问题：

1. Agent 是否需要本地文件或工具？
2. 对话是否多步骤？
3. 客户端是否需要中介工具调用？
4. 多个客户端或 Agent 是否会从共享协议中受益？

如果大部分答案是 yes，ACP 值得认真考虑。

## [m8-scenario] 公开示例：CampusGuide 回顾

假设一个团队在做 CampusGuide，一个虚构的校园问答助手。他们希望 Agent 能检查本地文档、运行验证脚本、并给出修改建议。

这是一个合理的 ACP 场景：本地文件重要、工具调用重要、客户端应该控制 Agent 能访问什么。

但如果这个团队只是想在网页里粘贴一段话让它改写，ACP 就没必要。

::: checkpoint
用决策四问快速判断：一个“把网页表单里的文本发给大模型润色”的功能，是否值得用 ACP？

--- answer
不值得。它不需要本地文件/工具，不是多步骤会话，不需要客户端中介工具调用，也不会从共享客户端- Agent 协议中明显受益。普通 HTTP API 更简单。
---
:::

## [m8-conclusion] 结论

ACP 是一种标准化工具。它只有在“统一的客户端- Agent 边界”能真正减轻集成痛苦时才有价值。

选择它，是因为它简化了系统；跳过它，是因为它只增加协议仪式而没有改善学习或开发流程。

## [m8-summary] 模块小结

- ACP 最强的地方在本地、多步骤、会用工具的 Agent 会话。
- 简单的单轮聊天通常不需要 ACP。
- 公开教学示例应该使用虚构、安全的项目。

::: exercise
你接到了一个需求：为内部开发者做一个命令行工具，让它可以调用本地 Agent 来审查代码、运行测试并把结果写回文件。判断 ACP 是否合适。

- A. 不合适，直接调用模型 API 就够了。
- B. 合适，因为涉及本地文件、工具调用、多步骤会话和共享协议收益。
- C. 应该用 A2A，因为需要“代码审查 Agent”和“测试 Agent”两个角色。
- D. 应该用 MCP 替代 ACP，因为 MCP 也能调用测试命令。

--- answer
**B** 最合适。该场景符合决策四问：本地文件/工具、多步骤、客户端中介工具调用、共享协议收益。D 的 MCP 只解决 Agent-工具层，不解决客户端- Agent 驱动层；C 的 A2A 在没有两个独立 Agent 角色时是过度设计；A 忽略了工具调用和本地上下文需求。
---
:::
