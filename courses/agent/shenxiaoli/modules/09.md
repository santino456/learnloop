## [m9-frontend] 前端层参考

### React 18

React 是用于构建用户界面的声明式组件库。深小理用它构建聊天界面、会话列表和消息流。

> 来源：[React 官方文档 — Thinking in React](https://react.dev/learn/thinking-in-react)、[State: A Component's Memory](https://react.dev/learn/state-a-components-memory)、[React 18 发布说明](https://react.dev/blog/2022/03/29/react-v18)。

**核心抽象**

| 概念 | 定义 | 为什么重要 |
|------|------|-----------|
| 组件 | 接收 props、管理 state、返回 UI 描述的函数 | 把界面拆成可复用、可测试的单元 |
| JSX | JavaScript 的语法扩展，允许在 JS 里写标签 | 让 UI 结构和渲染逻辑写在一起 |
| 单向数据流 | 数据自顶向下传递，子组件通过回调影响父组件 | 状态变化可预测，调试路径清晰 |
| 虚拟 DOM | React 维护的内存中 UI 表示，diff 后最小化真实 DOM 操作 | 减少直接操作 DOM 的性能开销和错误 |

**Hooks（函数组件的状态与副作用）**

| Hook | 用途 | 典型误用 |
|------|------|---------|
| `useState` | 组件内部状态 | 把不该触发重渲染的数据放进 state |
| `useEffect` | 同步外部系统（订阅、DOM 操作、数据获取） | 没写依赖数组或 cleanup 函数，导致重复执行或内存泄漏 |
| `useReducer` | 复杂状态逻辑 | 简单状态强行用 reducer，增加认知负担 |
| `useRef` | 保存不触发渲染的引用 | 用 ref 代替 state 做 UI 响应 |
| `useMemo` / `useCallback` | 缓存计算和函数引用 | 过早优化，反而增加比较开销 |

**React 18 并发特性**

- **Concurrent Rendering**：React 可以在渲染过程中中断和恢复，优先响应高优先级交互（如用户输入）。
- **Transitions**：用 `useTransition` 标记低优先级更新，避免阻塞 UI。
- **Suspense**：声明式异步边界，让组件在数据准备好之前显示 fallback。
- **Automatic Batching**：多个 setState 合并为一次渲染，减少重绘。

深小理里，消息流持续更新时用户仍能输入，靠的就是并发渲染把输入响应放在高优先级。

### Vite

Vite 是下一代前端构建工具，核心思路是**开发时用原生 ESM，生产时用 Rollup 打包**。

> 来源：[Vite 官方文档 — Why Vite](https://vitejs.dev/guide/why.html)、[Features](https://vitejs.dev/guide/features.html)。

| 阶段 | 做法 | 效果 |
|------|------|------|
| 开发 | 浏览器直接请求 ESM 模块，按需编译 | 冷启动快，无论项目多大都是秒级 |
| 预构建 | 用 esbuild 把 CommonJS 依赖转成 ESM | 解决浏览器原生 ESM 对 CJS 的支持问题 |
| 生产 | Rollup 打包、代码分割、Tree Shaking | 产物小、加载快 |
| HMR | 模块级热替换，只更新变化的部分 | 改代码后页面即时反馈 |

与 Webpack 的关键差异：Webpack 在开发时也要先打包整个依赖图，项目大了启动慢；Vite 跳过这一步，依赖预构建一次后，源码按请求编译。

深小理选 Vite，是因为聊天应用开发阶段需要频繁调试 UI 和 SSE 交互，HMR 速度直接影响效率。

### Tailwind CSS

Tailwind 是 utility-first CSS 框架。不写 `.chat-box { padding: 1rem; }`，而是写 `class="p-4"`。

> 来源：[Tailwind CSS 官方文档 — Utility-First Fundamentals](https://tailwindcss.com/docs/utility-first)、[Just-in-Time Mode](https://v2.tailwindcss.com/docs/just-in-time-mode)。

**设计系统内建值**

| 类别 | 示例类名 | 对应 CSS |
|------|---------|---------|
| 间距 | `p-4`, `m-2`, `gap-3` | padding、margin、gap 的预设步进值 |
| 颜色 | `bg-blue-500`, `text-gray-700` | 设计系统色板 |
| 布局 | `flex`, `grid`, `items-center` | Flexbox/Grid 常用组合 |
| 响应式 | `md:flex-row` | 在 `md` 断点下应用样式 |
| 状态 | `hover:bg-blue-600` | 伪类样式 |

**JIT 模式（Just-in-Time）**

- 开发时扫描源码，只生成实际用到的类。
- 支持任意值：`w-[123px]`、`bg-[#1da1f2]`。
- 产物大小与使用的类数量相关，而不是框架总类数。

**tradeoffs**

- 优点：开发快、风格一致、产物可控。
- 缺点：HTML 里 class 字符串长；复杂动画和伪元素仍需要自定义 CSS。

### TypeScript

TypeScript 是 JavaScript 的超集，增加静态类型系统。

> 来源：[TypeScript 官方文档 — The TypeScript Handbook](https://www.typescriptlang.org/docs/handbook/intro.html)。

**核心能力**

- 在编译期发现类型错误，而不是运行时。
- 为函数、对象、API 响应定义接口。
- 通过 `.d.ts` 文件描述第三方库类型。

**深小理建议开启的选项**

```json
{
  "compilerOptions": {
    "strict": true,
    "noImplicitAny": true,
    "strictNullChecks": true,
    "esModuleInterop": true
  }
}
```

- `strict`: 开启所有严格类型检查。
- `noImplicitAny`: 不允许隐式 `any`。
- `strictNullChecks`: `null` 和 `undefined` 必须显式处理。

深小理里，SSE 消息类型、API 响应结构、会话/消息模型都应定义接口。这样后端字段变化时，前端构建会报错，而不是到运行时才出现 `undefined`。

## [m9-backend] 后端层参考

### FastAPI

FastAPI 是 Python 的异步 Web 框架，基于 Starlette（ASGI）和 Pydantic（数据校验）。

> 来源：[FastAPI 官方文档 — Tutorial - User Guide](https://fastapi.tiangolo.com/tutorial/)、[Concurrency and async / await](https://fastapi.tiangolo.com/async/)。

**核心机制**

| 机制 | 说明 | 深小理用法 |
|------|------|-----------|
| Pydantic 模型 | 用 Python 类型注解定义请求/响应结构 | `ChatRequest`、`UserLogin` 等模型自动校验 |
| 依赖注入 | `Depends()` 抽离可复用逻辑 | JWT 校验、数据库连接、quota 检查 |
| `async def` | 路由函数异步执行 | 数据库查询、LLM 调用不阻塞线程 |
| `StreamingResponse` | 流式 HTTP 响应 | SSE 把 Agent 输出实时推给前端 |
| 自动 OpenAPI | 访问 `/docs` 生成交互式文档 | 前后端联调、测试接口 |

**SSE 响应的最小形态**

```python
from fastapi import FastAPI
from fastapi.responses import StreamingResponse

app = FastAPI()

@app.post("/chat")
async def chat(request: ChatRequest):
    async def event_stream():
        async for chunk in agent.run_stream(request.message):
            yield f"data: {chunk}\n\n"
    return StreamingResponse(
        event_stream(),
        media_type="text/event-stream"
    )
```

**为什么不是 Flask 或 Django？**

- Flask 偏同步，做 SSE 需要额外插件；FastAPI 的 `StreamingResponse` 是内建的。
- Django 太重型，ORM、admin、form 在深小理这种 API-first + Agent 的项目里是负担。
- 深小理的 Agent 层是 Python（LangGraph），后端统一语言降低上下文切换。

### Python 3.11

Python 3.11 是深小理后端的运行版本。

> 来源：[Python 官方文档 — What's New In Python 3.11](https://docs.python.org/3.11/whatsnew/3.11.html)。

**主要改进**

- **性能**：CPython 解释器优化，整体提升 10–60%，async 路径受益明显。
- **错误信息**：traceback 更精确，能指出表达式中具体出错的位置。
- **`TaskGroup`**：更优雅地管理多个并发任务，一个失败可以取消其余任务。
- **`ExceptionGroup`**：把多个异常打包抛出，适合并发场景的错误处理。

深小理里并发点多：SSE 连接、数据库查询、LLM 流式调用。Python 3.11 的性能和错误信息改善都有实际价值。

### JWT 认证

JWT（JSON Web Token）是无状态认证机制。用户登录后服务端签发 token，后续请求带上 token 即可验证身份。

> 来源：[RFC 7519 — JSON Web Token (JWT)](https://datatracker.ietf.org/doc/html/rfc7519)、[OWASP JWT Cheat Sheet](https://cheatsheetseries.owasp.org/cheatsheets/JSON_Web_Token_for_Java_Cheat_Sheet.html)。

**结构**

```text
header.payload.signature
```

- **header**：指定签名算法（如 HS256、RS256）。
- **payload**：携带声明（claims），如 `sub`（用户 ID）、`exp`（过期时间）、`iat`（签发时间）。
- **signature**：用服务端密钥对前两部分签名，防止篡改。

**深小理里的安全实践**

| 项目 | 做法 | 原因 |
|------|------|------|
| 密钥管理 | 放在环境变量，不硬编码 | 防止代码泄露导致 token 可被伪造 |
| 过期时间 | access token 15–60 分钟，配合 refresh token | 缩短泄露窗口 |
| 算法验证 | 服务端显式指定允许的算法 | 防止 `alg: none` 攻击 |
| 敏感声明 | 不放权限等敏感信息在 payload | token 一旦泄露就泄露这些声明 |

**前端存储位置 tradeoffs**

| 存储位置 | 优点 | 风险 |
|---------|------|------|
| 内存 | 最安全，XSS 也拿不到 | 刷新页面丢失，需重新登录 |
| localStorage | 刷新不丢，实现简单 | XSS 可读取 |
| httpOnly cookie | XSS 拿不到 | 需处理 CSRF，跨域配置复杂 |

深小理如果走安全优先，推荐 httpOnly cookie；如果走实现简单，可用 localStorage 但需严格控制内联脚本和 XSS 防护。

### PostgreSQL 14

PostgreSQL 是开源关系型数据库。深小理用它存用户、会话、消息、记忆、quota 等。

> 来源：[PostgreSQL 14 官方文档 — JSON Types](https://www.postgresql.org/docs/14/datatype-json.html)、[Indexes](https://www.postgresql.org/docs/14/indexes.html)。

**为什么选 PostgreSQL？**

| 能力 | 说明 | 深小理用途 |
|------|------|-----------|
| ACID 事务 | 完整支持原子性、一致性、隔离性、持久性 | quota 扣减、消息写入必须可靠 |
| JSONB | 二进制 JSON，支持索引和复杂查询 | messages 字段灵活存储 |
| 扩展生态 | PostGIS、pgvector 等 | 未来如需向量检索可直接扩展 |
| 连接稳定性 | 成熟可靠，裸机部署简单 | 减少运维复杂度 |

**JSONB 关键事实**

- JSONB 存储的是解析后的二进制格式，查询时比 JSON 文本更快。
- 可以用 GIN 索引加速 JSONB 路径查询：

```sql
CREATE INDEX idx_messages_content ON messages
USING GIN (content jsonb_path_ops);
```

- JSONB 不适合存大文件内容；图片/文件应走 OSS。
- 高频过滤字段（如 `user_id`、`session_id`）应抽成普通列并建 B-tree 索引，比在 JSONB 里查快。

### psycopg 3

psycopg 是 PostgreSQL 的 Python 驱动。psycopg 3 是新一代版本。

> 来源：[psycopg 3 官方文档 — Basic usage](https://www.psycopg.org/psycopg3/docs/basic/usage.html)、[Async support](https://www.psycopg.org/psycopg3/docs/advanced/async.html)。

**psycopg 3 相对 2 的主要变化**

- 原生支持异步：`psycopg.AsyncConnection`、`psycopg.AsyncCursor`。
- 更好的类型转换和参数处理。
- 支持 `Pipeline` 模式批量执行。

**深小理为什么直接用它而不是 SQLAlchemy？**

- 复杂查询和性能调优时，ORM 生成的 SQL 不透明。
- 直接写 SQL 可以精确控制索引使用、批量插入、锁行为。
- 但代价是：必须手动管理连接池、手动写参数化查询防止 SQL 注入。

**参数化查询示例**

```python
# 安全：数据库驱动把参数当数据处理
await cur.execute("SELECT * FROM users WHERE id = %s", (user_id,))

# 危险：字符串拼接，存在 SQL 注入
await cur.execute(f"SELECT * FROM users WHERE id = {user_id}")
```

## [m9-agent] Agent 层参考

### LangGraph

LangGraph 是 LangChain 生态中的 Agent 编排框架。它把 Agent 执行过程建模成**有向状态图**。

> 来源：[LangGraph 官方文档 — Concepts](https://langchain-ai.github.io/langgraph/concepts/)、[How-to Guides](https://langchain-ai.github.io/langgraph/how-tos/)。

**核心抽象**

| 概念 | 说明 |
|------|------|
| State | 整个图共享的状态对象，通常包含 `messages`、`loop_count` 等 |
| Node | 一个执行步骤，如 `prepare`、`model`、`tools` |
| Edge | 节点之间的转移；可以是固定跳转或条件路由 |
| Conditional Edge | 根据状态决定下一步走哪个节点 |
| Checkpoint | 图执行到某一步的快照，支持中断后恢复 |
| Interrupt | 暂停执行等待外部输入，如用户确认隐私操作 |

**为什么不用自己写循环？**

- 自研 ReAct 循环容易把状态、错误处理、工具调用混在一起。
- LangGraph 把循环显式化成图，便于单步调试、加节点、改路由。
- 中断/恢复机制内建，适合需要人工确认的隐私场景。

### ReAct 模式

ReAct = Reasoning（推理）+ Acting（行动）。

> 来源：[ReAct: Synergizing Reasoning and Acting in Language Models](https://arxiv.org/abs/2210.03629)（Yao et al., 2022）。

基本循环：

```text
Thought: 用户问的是... 我需要先...
Action: 调用 search_kb(query="...")
Observation: 搜索结果返回 ...
Thought: 根据结果，下一步应该...
Final Answer: ...
```

在深小理的 LangGraph 实现里，这对应：

- `prepare`：整理上下文和系统提示。
- `model`：调用 LLM，决定生成回答或调用工具。
- `tools`：执行模型选择的工具。
- `should_continue`：判断是继续循环还是结束，通常检查 `loop_count` 是否超限。

### LLM API 与流式

深小理使用 OpenAI 兼容 API，意味着接口格式遵循 OpenAI 的 `/chat/completions` 规范。

> 来源：[OpenAI API 文档 — Chat Completions](https://platform.openai.com/docs/guides/chat-completions)、[Function Calling](https://platform.openai.com/docs/guides/function-calling)。

**关键参数**

| 参数 | 作用 | 典型设置 |
|------|------|---------|
| `model` | 指定模型 ID | `gpt-4o`、`gpt-3.5-turbo` 或兼容接口的模型名 |
| `messages` | 对话历史 | system / user / assistant / tool 消息列表 |
| `temperature` | 控制随机性 | 0–2；事实提取用 0，创意生成用 0.7–1 |
| `max_tokens` | 最大生成长度 | 根据场景设置，避免截断或浪费 |
| `stream` | 是否流式返回 | `true` 时通过 SSE 逐字返回 |
| `tools` | 可调用工具列表 | 每个工具包含名称、描述、参数 schema |

**函数调用（Tool Calling）流程**

1. 请求里带 `tools` 参数，描述可用工具。
2. 模型决定调用某个工具时，返回 `tool_calls` 而不是普通内容。
3. 后端执行工具，把结果以 `tool` 角色的消息加回 `messages`。
4. 再次调用模型，模型基于工具结果生成最终回答。

### 工具注册

深小理的工具注册机制让 Agent 动态发现可调用的能力。

典型实现模式：

```python
@tool_registry.register(description="搜索知识库")
def search_kb(query: str) -> list[dict]:
    """根据 query 搜索知识库并返回相关片段。"""
    ...
```

LangChain 会自动把函数签名转成 JSON Schema：

```json
{
  "name": "search_kb",
  "description": "搜索知识库",
  "parameters": {
    "type": "object",
    "properties": {
      "query": {"type": "string", "description": "搜索关键词"}
    },
    "required": ["query"]
  }
}
```

模型靠 `description` 和参数 schema 决定什么时候调用、传什么参数。

## [m9-storage] 存储层参考

### PostgreSQL JSONB

JSONB 是 PostgreSQL 的二进制 JSON 存储类型。

> 来源：[PostgreSQL 14 官方文档 — JSON Types](https://www.postgresql.org/docs/14/datatype-json.html)。

**JSONB vs JSON（PostgreSQL 里两种 JSON 类型）**

| 特性 | JSON | JSONB |
|------|------|-------|
| 存储格式 | 原始文本 | 解析后的二进制 |
| 查询速度 | 慢（每次解析） | 快 |
| 索引支持 | 不支持 | 支持 GIN 索引 |
| 空格/重复键 | 保留 | 不保留 |
| 写入速度 | 快 | 稍慢（需解析） |

深小理用 JSONB 存 messages，因为需要查询和索引能力。

**GIN 索引示例**

```sql
CREATE INDEX idx_messages_content_type ON messages
USING GIN (content jsonb_path_ops);
```

适用场景：按 JSONB 内部字段过滤或搜索。

### 阿里云 OSS

OSS（Object Storage Service）是对象存储服务，适合存图片、文件、备份等大对象。

> 来源：[阿里云对象存储文档](https://help.aliyun.com/product/31815.html)。

**核心概念**

| 概念 | 说明 |
|------|------|
| Bucket | 存储空间，全局唯一命名 |
| Object | 存储的文件对象，由 key 唯一标识 |
| Endpoint | 访问域名，分公网/内网/加速域名 |
| ACL | 访问权限：私有、公共读、公共读写 |
| Presigned URL | 临时签名链接，用于私有 bucket 临时授权访问 |

**深小理用法**

- 用户上传图片 → 后端把图片推送到 OSS → 返回图片 URL 给前端显示。
- 避免把图片存在应用服务器本地或数据库里，减轻带宽和存储压力。

**安全注意**

- Bucket 默认设为私有，需要访问时生成带签名的临时 URL。
- 配置防盗链（Referer 白名单），防止其他站点盗用流量。
- 上传接口限制文件大小和类型，防止恶意文件占用存储。
- AccessKey ID/Secret 必须放在服务端，绝不能写进前端代码。

## [m9-deploy] 部署层参考

### Nginx

Nginx 是高性能 Web 服务器和反向代理。深小理用它接收外部请求并转发给 FastAPI。

> 来源：[Nginx 官方文档](https://nginx.org/en/docs/)、[Module ngx_http_proxy_module](https://nginx.org/en/docs/http/ngx_http_proxy_module.html)。

**核心职责**

| 职责 | 说明 |
|------|------|
| 反向代理 | 把 80/443 端口请求转发给后端（如 8000 端口） |
| 静态文件 | 直接服务前端构建产物 |
| SSL 终止 | 处理 HTTPS 证书，后端只跑 HTTP |
| 负载均衡 | 多后端实例时分发请求 |
| 限流/压缩 | 控制请求速率、启用 gzip |

**SSE 关键配置**

SSE 是 HTTP 长连接，Nginx 默认会缓冲响应，导致前端无法实时收到数据。必须关闭缓冲：

```nginx
location /chat {
    proxy_pass http://127.0.0.1:8000;
    proxy_http_version 1.1;
    proxy_set_header Connection "";
    proxy_buffering off;
    proxy_cache off;
    proxy_read_timeout 3600s;
}
```

关键指令：

- `proxy_http_version 1.1`：HTTP/1.1 支持 keepalive。
- `proxy_buffering off`：不缓冲响应，实时推流。
- `proxy_read_timeout 3600s`：SSE 连接可以保持 1 小时不被断开。

### systemd

systemd 是 Linux 系统和服务管理器。深小理用它把后端进程注册为系统服务。

> 来源：[systemd 官方文档 — systemd.service](https://www.freedesktop.org/software/systemd/man/latest/systemd.service.html)、[systemd-journald.service](https://www.freedesktop.org/software/systemd/man/latest/systemd-journald.service.html)。

**典型 unit 文件**

```ini
[Unit]
Description=ShenXiaoLi API
After=network.target postgresql.service

[Service]
Type=notify
User=shenxiaoli
WorkingDirectory=/opt/shenxiaoli
ExecStart=/opt/shenxiaoli/venv/bin/python -m uvicorn main:app --host 127.0.0.1 --port 8000
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target
```

**常用命令**

```bash
sudo systemctl start shenxiaoli    # 启动服务
sudo systemctl stop shenxiaoli     # 停止服务
sudo systemctl restart shenxiaoli  # 重启服务
sudo systemctl status shenxiaoli   # 查看状态
sudo journalctl -u shenxiaoli -f   # 实时查看日志
```

**`Restart=on-failure` 含义**

- 服务异常退出时自动重启。
- 正常 `systemctl stop` 不会触发重启。
- 配合 `RestartSec=5` 避免崩溃后疯狂重试。

### 裸机部署 vs 容器化

深小理用 Nginx + systemd 裸机部署。下面是不同部署方式的边界。

> 来源：本地 m7《部署与运维：从本机到线上》。

| 维度 | 裸机 + systemd | Docker Compose | Kubernetes |
|------|---------------|----------------|------------|
| 适用规模 | 单体应用 | 几个服务 | 几十个服务 |
| 环境一致性 | 依赖服务器状态 | 镜像一致 | 强一致 |
| 启动速度 | 快 | 中等 | 慢 |
| 回滚能力 | 需脚本 | 简单 | 原生支持 |
| 排查难度 | 低（journalctl） | 中等 | 高 |
| 资源隔离 | 弱 | 中等 | 强 |

深小理当前阶段是单体应用，裸机部署路径最短。如果未来拆出独立服务（如单独的图片服务、搜索服务、记忆服务），就应评估 Docker Compose 或 K8s。
