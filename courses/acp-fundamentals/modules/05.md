## [m5-three-lines] Three protocol lines

ACP, MCP, and A2A describe different relationships.

- ACP: a client drives an agent.
- MCP: an agent accesses tools or data sources.
- A2A: one agent coordinates with another agent.

## [m5-acp] ACP: client to agent

ACP answers: how does the user-facing client talk to the agent that will do the work?

It is most relevant when the session is interactive, local context matters, and the agent may need tools.

## [m5-mcp] MCP: agent to tools

MCP answers: how does the agent safely and consistently access external tools, files, APIs, or data?

An ACP client does not need to expose every MCP detail to the user. It may simply route tool calls and return results.

## [m5-a2a] A2A: agent to agent

A2A answers: how do multiple agents discover, delegate, and exchange results?

Use it when the system genuinely has multiple agent roles. Do not add it just because a single agent has multiple steps.

## [m5-choose] Choosing the layer

Ask which boundary is painful:

- Client integration painful: consider ACP.
- Tool and data integration painful: consider MCP.
- Multi-agent handoff painful: consider A2A.

## [m5-summary] Module recap

- ACP connects clients and agents.
- MCP connects agents and tools.
- A2A connects agents with other agents.
- Good architecture starts by naming the boundary, then choosing the protocol.
