## [m4-asic] 为什么用 AI ASIC 超节点

::: concept
title: 全栈基于自研 AI ASIC
why: 基础设施选择会影响并行策略、内存优化和故障恢复设计。

LongCat-2.0 的完整训练与大规模部署完全基于 **AI ASIC 超节点**。与成熟的 Nvidia GPU 生态相比，AI ASIC 的软件社区仍在发展中，因此团队投入大量精力构建稳定、安全、可扩展的基础设施。

同类实践：Google TPU Pod、AWS Trainium 集群、华为昇腾集群也走自研加速器路线，核心差异在于软件栈成熟度。

## [m4-training] 训练优化

::: concept
title: 确定性与可靠性
why: 数万卡规模训练必须可复现、可自动恢复。

为保障生产环境的可复现性，团队在通信和计算路径上都强制确定性：

- 自研确定性算子和模块覆盖 Embedding、Flash Attention、LSA、MoE 层。
- 所有 reduction 类算子采用二叉树分段累加，减少浮点误差累积。
- 在真实 LLM 负载下对加速器计算精度进行严格高精度基线验证。
- 在部分计算密集型算子中引入 bit-flip 检测，及时捕获硬件位翻转异常。
- 端到端监控驱动故障识别、流量切换与自动恢复；隔离故障链路对训练无感知影响，修复后需通过压力测试才重新加入。

参考：大规模训练可靠性可参考 [GPT-4 训练稳定性经验](https://arxiv.org/abs/2303.08774) 与 DeepSeek-V3 训练报告。

Source: [LongCat-2.0 官方博客](https://longcat.chat/blog/longcat-2.0/)
:::

::: concept
title: 6D 并行与内存优化
why: AI ASIC 单卡显存显著小于 H800，内存是主要瓶颈。

扩展策略：

- **6D 并行**：在标准 TP/CP/EP/DP/PP 基础上，增加 **EMBP** 来并行和加速 N-gram Embedding。
- **Superpod**：每个物理 Superpod 最多 48 台机器，内部全互联高带宽，之间通过 RoCE 互联；相比同规模 GPU 环境带来约 **30% 预训练吞吐提升**。
- **内存优化**：ZeRO-1、选择性重计算、OOM-aware offloading、将 padding token 路由到 zero-expert。
- **Muon 优化器**：在大规模 AI ASIC 上部署，并针对 TP 并行、DP 状态冗余去除和对称矩阵乘核做了优化。

延伸阅读：
- [ZeRO: Memory Optimizations Toward Training Trillion Parameter Models](https://arxiv.org/abs/1910.02054)
- [Muon is Scalable for LLM Training](https://arxiv.org/abs/2502.16982) / [GitHub](https://github.com/KellerJordan/Muon)

Source: [LongCat-2.0 官方博客](https://longcat.chat/blog/longcat-2.0/)
:::

::: concept
title: 长上下文训练
why: 1M token 训练需要专门的并行与通信设计。

- **LSA 算子与前向优化**：实现自研确定性注意力算子，覆盖稠密 warmup 与稀疏阶段；采用前向-only 稠密 warmup 策略，一次前向同时计算 KL loss 和梯度。
- **1M context scaling**：基于 all-gather 的 CP 并行可扩展到 **512+**，支持原生 1M 长度训练；在 get-batch 阶段重排数据，并用 balanced CP 策略保持负载均衡。
- **计算-通信重叠**：shortcut-layer 架构让 MoE 通信与并行分支计算重叠，LSA top-k 索引计算与 KV all-gather 重叠。

参考：CP（Context Parallelism）在长上下文训练中的经典实现可参见 [Ring Attention](https://arxiv.org/abs/2310.01889)。

Source: [LongCat-2.0 官方博客](https://longcat.chat/blog/longcat-2.0/)
:::

## [m4-inference] 推理优化

::: concept
title: 服务 1.6T/1M 模型的挑战
why: 解码阶段受 HBM 容量、HBM I/O 和节点间互联带宽三重约束。

LongCat-2.0 从模型、设备和部署三个层面优化推理：

- **注意力**：prefill 与 decode 阶段都采用 absorb 计算模式；将 indexer 与 MLA prolog 在并发流上流水线化；用 KV-cache Parallelism（KVP）分片 KV-cache。
- **ScMoE**：在 LongCat-Flash 计算-通信重叠基础上，通过加速器显式 per-core 控制实现 dense 与 MoE 分支完全并行。
- **Super Kernel**：用超核减少图模式下的 intra-kernel 启动开销。
- **Weight Prefetch**：利用较大 L2 缓存预取权重，隐藏前一算子的 I/O 延迟。
- **Scale Up/Out**：P 节点与 D 节点间 KV-cache 传输使用加速器内置 200 Gbps 网卡；KV-cache store 用 host RDMA 网卡构建；TP/SP/KVP 在 scale-up 域内完成。

参考：PD 分离与 KV-cache 分片策略可参见 [DistServe](https://arxiv.org/abs/2401.09670) 与 vLLM 的 PagedAttention。

Source: [LongCat-2.0 官方博客](https://longcat.chat/blog/longcat-2.0/)
:::

::: concept
title: PD 分离与并行策略
why: 预填充和解码的瓶颈不同，需要分别优化。

- **Prefill 节点**：长序列受节点间通信带宽约束，MoE dispatch/combine 占主导；使用多节点 chunked pipeline parallelism（CPP）缩小 EP 域，内部用 Attention Sequence Parallelism（SP）缓解长序列计算压力。
- **Decode 节点**：瓶颈是显存与 KV-cache I/O；使用 KVP 分片 KV-cache，并采用大 EP 度（EP128）降低单卡权重与 expert I/O。
- **EPLB**：Decode 端大 EP 度容易负载不均，通过 Expert-Parallel Load Balancing 解决；统计与放置计算异步进行，不阻塞前向。

Source: [LongCat-2.0 官方博客](https://longcat.chat/blog/longcat-2.0/)
:::

## [m4-exercise] 练习：匹配优化与瓶颈

::: exercise
把下列优化与对应的主要瓶颈连线。

- KVP
- EP128
- CPP
- Weight Prefetch
- EMBP

瓶颈：

1. 单卡 KV-cache 容量不足
2. 解码阶段单卡权重/专家 I/O 过大
3. 预填充阶段长序列通信带宽不足
4. 加速器 HBM 带宽有限但 L2 大
5. N-gram Embedding 需要额外并行维度

--- answer
- KVP → 1
- EP128 → 2
- CPP → 3
- Weight Prefetch → 4
- EMBP → 5
---
:::
