## [m2-schema] The Unified Environment Trajectory Schema

Training a single world model across seven domains with state representations as varied as file-system snapshots for Terminal and UI view hierarchies for Android requires a shared format. Qwen-AgentWorld adopts a unified environment trajectory schema uniformly across all seven domains and training stages.

::: concept
title: Unified Environment Trajectory Schema

**Unified Environment Trajectory Schema**

system_prompt := task_description ⊕ action_space ⊕ initial_state ⊕ demonstrations ⊕ simulation_instruction

turn_t := (action_t, observation_t)

trajectory := system_prompt ⊕ [turn_1, …, turn_T]

Source: [Qwen-AgentWorld](https://arxiv.org/abs/2606.24597), §2.2
:::

## [m2-components] Five Components of the System Prompt

The system prompt has five components, whose construction is detailed in §3.1.2.

::: figure
src: assets/qwen-agentworld-fig-3.png
alt: Figure 3 anatomy of a Terminal domain LWM RL system prompt showing five components with static components in blue and dynamic in red.
caption: Figure 3: Anatomy of a Terminal domain LWM RL system prompt, showing the five components defined in §2.2. Blue = static (shared across trajectories); red = dynamic (injected per trajectory).
:::

1. **Task description**: Instructs the model to act as a world model for a specific domain and defines the simulation objective.
2. **Action space**: Enumerates the available tools or operations and their calling conventions.
3. **Initial state**: Specifies the environment’s starting configuration before any interaction begins (installed packages, file-system layout, UI screen state, or other preconditions).
4. **Demonstrations**: Optional few-shot (action, observation) examples.
5. **Simulation instruction**: Specifies controllable simulation conditions (e.g., "hide the answer from the web_search responses").

Source: [Qwen-AgentWorld](https://arxiv.org/abs/2606.24597), §2.2, §3.1.2

::: concept
title: Static vs. dynamic components

**Static vs. dynamic components**: For all domains except MCP and SWE, the action space and demonstrations are predefined (static). MCP and SWE trajectories require per-trajectory action spaces because the available tools differ across MCP server instances and code repositories. The initial state and simulation instruction are dynamic (filled per trajectory) when present.

Source: [Qwen-AgentWorld](https://arxiv.org/abs/2606.24597), §3.1.2
:::

## [m2-expansion] Trajectory-to-Turn Expansion and Filtering

Raw agentic trajectories from heterogeneous sources are normalized into the unified schema through domain-specific handlers and shared preprocessing steps.

::: flow
Trajectory with T turns
  → Expand into T turn-level prediction samples
    → Each turn t uses (turn_1, …, turn_{t-1}) + state/action at t as input
    → Observation at turn t as target
  → Randomly sample one turn per trajectory for training split
  → Apply data filtering (retry-cycle skipping, no-change turn filtering)
  → Construct system prompt with five components
  → Final training sample
:::

::: concept
title: Turn-level expansion

**Turn-level expansion**: Because agent–environment interaction is inherently multi-turn and each observation depends only on its preceding history, every turn within a trajectory is itself a valid prediction instance once paired with its prior context. Because each trajectory is expanded into turn-level prediction samples, the model receives supervision at every turn.

Source: [Qwen-AgentWorld](https://arxiv.org/abs/2606.24597), §3.1.2
:::

::: concept
title: Retry-cycle skipping

**Retry-cycle skipping**: Agents frequently fall into "garbage output → error → retry" cycles. Simply deleting these turns breaks the state chain. Instead, the offending (user, assistant) pairs are skipped while preserving the current state so that the next valid turn inherits the correct history.

Source: [Qwen-AgentWorld](https://arxiv.org/abs/2606.24597), §3.1.2
:::

::: concept
title: No-change turn filtering (GUI domains)

**No-change turn filtering (GUI domains)**: Turns whose pre-action and post-action states show no effective change are removed. If retained, these samples teach the model to copy the previous state as-is regardless of the action taken, undermining its ability to predict how actions change the environment.

Source: [Qwen-AgentWorld](https://arxiv.org/abs/2606.24597), §3.1.2
:::

## [m2-exercise] Exercise: Construct a System Prompt

**Task**: Given the following scenario, construct the five components of a unified environment trajectory schema for the Terminal domain.

- Scenario: A developer wants to simulate a Python data-processing pipeline inside a Docker container.
- The container has Python 3.10, pandas 1.5, and a /data directory with CSV files.
- The developer will run scripts that read CSVs, perform group-by operations, and write results to /output.
- You want the simulator to occasionally inject a "disk full" error during the write phase to test the agent’s error handling.

**Instructions**: Write the five components (task description, action space, initial state, demonstrations, simulation instruction) following the schema in §2.2. Mark each component as static or dynamic.

**Verification criteria**: The initial state must include installed packages and file-system layout. The simulation instruction must specify the controllable error condition. The action space must enumerate the relevant operations (bash commands, Python script execution).

Source: [Qwen-AgentWorld](https://arxiv.org/abs/2606.24597), §2.2, §3.1.2

---

**Module Summary**: The unified schema enables cross-domain generalization by representing all seven domains as a system prompt plus alternating (action, observation) turns. Trajectory-to-turn expansion provides supervision at every turn, while retry-cycle skipping and no-change filtering remove harmful samples.
