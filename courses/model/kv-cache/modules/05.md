## [m5-confusion] KV Cache 的分配难题

假设你在运行一个推理服务，同时处理多个不同长度的请求：

- 请求 A：prompt 1000 token，要生成 500 token
- 请求 B：prompt 200 token，要生成 2000 token
- 请求 C：prompt 500 token，要生成 100 token

如果为每个请求预分配最大可能长度的 KV Cache（比如 4096），大部分空间会被浪费。如果动态分配，频繁的内存分配和拷贝又会影响性能。更麻烦的是，请求完成后释放的内存块大小不一，很快就会产生碎片。

这就是 PagedAttention 要解决的问题。

## [m5-idea] 操作系统的分页思想

PagedAttention（Kwon et al., 2023, vLLM）借鉴了操作系统虚拟内存的分页机制：

1. 把 KV Cache 分成固定大小的**块（block）**，比如每块存 16 个 token 的 K 和 V。
2. 每个请求的 KV Cache 不需要连续存储，只需要一个**块表（block table）**记录逻辑块到物理块的映射。
3. 物理块从全局**块池（block pool）**中分配，用完回收。

```text
逻辑视图（请求 A）:  [token_0-15] [token_16-31] [token_32-47] ...
                       |            |            |
物理块表映射:         block_3      block_7      block_1
```

> 关键洞察：逻辑上连续的 KV Cache 在物理上可以不连续。这消除了外部碎片，也避免了预分配造成的浪费。

::: checkpoint
PagedAttention 的核心思想来自哪个领域的成熟技术？

--- answer
操作系统虚拟内存的分页机制。把 KV Cache 分成固定大小的块，通过块表映射逻辑块到物理块。
---
:::

## [m5-sharing] 块级别的共享

分页机制还带来一个额外好处：**块级共享**。当多个请求使用相同的 prompt 前缀时（比如多用户问同一个系统提示词），它们可以共享前缀对应的物理块，只在 diverge 的地方分配新块。

```text
请求 A: [shared_block_0] [shared_block_1] [A_unique_block_2]
请求 B: [shared_block_0] [shared_block_1] [B_unique_block_2]
```

这大幅减少了多并发下的显存占用，尤其在批量处理相似请求时效果显著。

::: checkpoint
PagedAttention 的块级共享在什么场景下收益最大？

--- answer
多个请求共享相同 prompt 前缀的场景，比如多用户使用相同的系统提示词，或批量处理相似输入。
---
:::

## [m5-copy-on-write] Copy-on-Write

当共享块中的某个请求需要修改自己的 KV Cache（比如生成新 token 追加到缓存），PagedAttention 使用**写时复制（Copy-on-Write）**：

1. 先共享物理块。
2. 当某个请求需要写入时，复制该块到新物理块，再修改。
3. 其他请求继续共享原块。

这保证了共享的安全性和效率。

## [m5-impact] 实际效果

依据：vLLM 论文（Kwon et al., 2023）中的实验显示，在相同显存预算下，PagedAttention 相比 Hugging Face 的 naive 实现，吞吐量提升了 2-4 倍。这种提升主要来自：

- 消除了预分配浪费。
- 消除了外部碎片。
- 实现了前缀块共享。

> 注意：具体的 2-4 倍数字来自 vLLM 论文的特定实验设置，实际收益取决于 workload 特征。

## [m5-limitations] 局限与考量

PagedAttention 不是万能的：

- **块大小选择**：块太小，块表开销大；块太大，内部碎片多。通常选择 16 或 32 token。
- **内核支持**：需要定制的 CUDA kernel 来支持非连续 KV Cache 的高效读取。vLLM 提供了这些 kernel 的实现。
- **不适用于所有场景**：对于单请求、短序列的推理，分页管理的 overhead 可能不值得。

## [m5-summary] 模块小结

- PagedAttention 把 KV Cache 分成固定大小的块，逻辑连续、物理不连续。
- 块表映射消除了外部碎片和预分配浪费。
- 块级共享和 Copy-on-Write 让多请求前缀复用成为可能。
- vLLM 的实验显示吞吐量可提升 2-4 倍，但具体收益取决于 workload。
- 块大小选择和定制 kernel 是实际部署中的关键考量。

::: checkpoint
PagedAttention 中，当共享块需要被修改时，使用什么机制保证其他请求不受影响？

--- answer
Copy-on-Write（写时复制）。修改前先复制一份新块，其他请求继续共享原块。
---
:::

::: checkpoint
PagedAttention 不适合以下哪种场景？

A. 多并发长序列推理  
B. 多请求共享相同前缀  
C. 单请求短序列推理  
D. 动态长度生成任务

--- answer
C。单请求短序列推理中，分页管理的 overhead 可能超过收益，简单的预分配或动态扩展反而更高效。
---
