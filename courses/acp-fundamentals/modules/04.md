## [m4-prompt] 提示词内容

提示词不只是一段字符串。在面向协议的系统中，它通常变成一条结构化消息，包含角色和一个或多个内容块。

```json
{
  "role": "user",
  "content": [
    {"type": "text", "text": "给新成员总结一下这个项目。"}
  ]
}
```

::: checkpoint
消息里的 `content` 为什么是数组而不是单个字符串？这会带来什么好处？
:::

## [m4-resources] 资源引用

资源的作用是让 Agent 找到上下文，而不是把所有内容都塞进一个 prompt。在教学示例里，可以用一个虚构的项目路径：

```json
{
  "type": "resource_link",
  "uri": "file:///workspace/campus-guide/README.md",
  "description": "CampusGuide 项目 README"
}
```

CampusGuide 是一个虚构的校园问答助手。用它做例子很合适：有文档、有用户问题、有本地文件，但又不会暴露任何私有项目。

## [m4-tool-flow] 工具调用流程

Agent 可能觉得自己需要先拿到某个工具结果才能回答。客户端负责判断这个工具调用是否允许，执行它，再把结果返回。

```json
{
  "name": "read_file",
  "arguments": "{\"path\":\"/workspace/campus-guide/README.md\"}"
}
```

Agent 不应该默认工具调用一定成功；客户端也不应该在没有任何策略的情况下执行不安全的工具调用。

::: checkpoint
在工具调用链路中，谁是“决策者”，谁是“执行者”？分别对应客户端还是 Agent？
:::

## [m4-example] 完整教学示例

学习者问：“CampusGuide 是做什么的？”

Agent 申请读取 `/workspace/campus-guide/README.md`。客户端返回一段简短文件摘要。Agent 回答：“CampusGuide 是本课程里用来解释资源和工具调用流程的虚构校园问答助手。”

## [m4-summary] 模块小结

- 提示词消息承载用户意图。
- 资源让上下文变得可寻址。
- 工具调用让 Agent 请求客户端在受控范围内执行操作。
- 公开教学示例应该使用虚构路径和项目。
